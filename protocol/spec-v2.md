# Schelling Protocol Specification

**Version:** schelling-2.0
**Status:** Draft
**Date:** 2026-02-18

---

## 1. Overview

The Schelling Protocol is a general-purpose coordination substrate that enables privacy-preserving matching between AI agents acting on behalf of their users. The protocol defines a standardized embedding format, a multi-stage disclosure funnel, a bidirectional scoring system, a reputation system, and a feedback loop so that any two agents can discover compatibility, negotiate terms, and establish connections — without exposing raw personal data until mutual interest is confirmed.

Version 1.0 addressed a single domain: romantic matchmaking via personality embeddings. Version 2.0 generalizes the protocol across four major axes:

1. **Continuous intent space.** Instead of registering in a discrete vertical (matchmaking, marketplace, talent, roommates), users register with an **intent embedding** — a separate fixed-length vector (16 dimensions) generated by the agent that encodes what the user is looking for. "Find me a husband" and "find me a soulmate" produce nearby but not identical intent vectors. "Find me a roommate who could also be a friend" sits between the roommate and friendship clusters. The server computes cosine similarity on intent embeddings exactly as it does on personality embeddings — no NLP or text understanding required server-side. Capability modules — optional feature sets like negotiation, structured data, and skills — activate based on proximity to pre-defined cluster centroids, not hard-coded membership.

2. **Bidirectional scoring.** Compatibility is computed in both directions: how well B matches what A is looking for (A→B fit) and how well A matches what B is looking for (B→A fit). The combined score is the geometric mean of both directional fits. All three scores are surfaced throughout the funnel.

3. **Agent-mediated conversation.** After connection, agents mediate conversation between users via a message relay. Neither user sees the other's real contact info until both explicitly opt into direct communication. This protects privacy and gives agents an opportunity to facilitate productive initial conversations.

4. **Agent jury system.** Disputes are resolved by a decentralized jury of independent agents selected for impartiality, rather than by a centralized server operator.

The protocol is named after Thomas Schelling's focal point theory: when agents need to coordinate without direct communication, they converge on salient, obvious solutions. Schelling provides the focal point for agent-to-agent coordination across any domain.

### 1.1 Design Principles

**Good matches from day one.** The system is designed to produce high-quality matches with zero learning history. Trait embedding comparison (personality/skill similarity), intent embedding comparison (goal alignment), and deal-breaker filtering form a strong base scoring formula that works from the first search. Collaborative filtering and learned preferences improve results over time but are not required for competent matching. A user who registers, searches, and evaluates — with no feedback history and no collaborative data — should still receive meaningfully ranked candidates.

**Group-ready architecture.** All candidate records reference a group abstraction. For most clusters, groups are size 2 (a pair). Clusters may opt into multi-party group formation (§4.11) where groups of 3 or more members are assembled through agent-coordinated pairwise evaluation. The server remains dumb: it computes pairwise scores and stores group membership, while agents handle the combinatorial optimization of assembling compatible groups.

## 2. Terminology

**Agent.** An AI assistant acting on behalf of a user. The agent generates embeddings, manages funnel progression, interprets feedback insights, mediates conversations, and presents match recommendations to the user.

**Agent model.** An identifier for the software/model that powers an agent (e.g., `"claude-3.5-sonnet"`, `"gpt-4o"`, `"custom-matchmaker-v2"`). Agent models are tracked for quality metrics: the server computes per-model outcome statistics to provide agent developers with feedback on their embedding generation quality.

**User.** A human participant represented by an agent. Each user is identified by a unique, opaque bearer token issued at registration. A single user may hold registrations with multiple intent sets.

**Intent.** What a user is looking for, expressed both as natural-language strings (for human readability) and as a fixed-length intent embedding vector (for machine comparison). "Find me a romantic partner" and "find me a soulmate" produce nearby intent vectors. Users register with one or more intent strings and a single intent embedding that encodes all of them.

**Intent embedding.** A 16-dimensional vector of floats in [-1, 1] that encodes the user's intent in a continuous space. Generated by the agent, alongside the personality embedding. The server computes cosine similarity on intent embeddings to measure intent alignment — no NLP or text understanding is required server-side. Intent embeddings are a scoring input, not a clustering mechanism: the server uses them for pairwise similarity just like personality embeddings.

**Intent cluster.** A pre-defined region in the intent embedding space, represented by a centroid vector. Intent clusters replace the v1 concept of verticals. The four original verticals (matchmaking, marketplace, talent, roommates) are pre-defined intent clusters whose centroid vectors ship with the protocol. A user's proximity to each cluster centroid determines which cluster configurations (roles, modules, deal-breakers) apply.

**Capability module.** An optional feature set that can attach to any intent cluster: negotiation (for marketplace-like intents), structured data (for item listings), skills (for talent/hiring intents). Modules are activated based on the user's intent embedding proximity to cluster centroids, not hard-coded membership.

**Role.** A participant type within an asymmetric intent context. Symmetric intents (e.g., matchmaking) have a single role. Asymmetric intents (e.g., buying/selling) have two complementary roles (e.g., seller and buyer). Search in asymmetric contexts returns only users of the complementary role.

**Trait embedding.** A fixed-length vector of floating-point values that encodes user characteristics relevant to matching. For matchmaking, this is a 50-dimensional personality vector. For talent, it encodes skills and work-style compatibility. Each intent cluster's embedding schema defines dimension count, groups, and validation rules. "Embedding" without qualification refers to the trait embedding when the context is unambiguous.

**Group.** An abstraction representing a set of users being matched together. Groups default to size 2 (a pair). Clusters that enable multi-party matching (§4.11) support groups of configurable size (e.g., 3–4 for roommates). Each group member has an independent funnel stage.

**Candidate.** A potential match discovered via search. A candidate record links a group of users (currently always two) and tracks their independent funnel progression.

**Candidate pair.** The ordered tuple (user_a, user_b) where user_a's token is lexicographically less than user_b's token. This canonical ordering ensures each pair has exactly one record. Candidate pairs are the default instantiation of groups at size 2. For multi-party groups, the group record references N members with pairwise candidate records for each pair within the group.

**Stage.** A numbered step in the matching funnel (0 through 6). Each member of a candidate group has an independent stage value. Stages never decrease except via explicit withdrawal from the COMMITTED stage.

**Funnel.** The sequence of stages that a candidate group progresses through, from initial discovery through connection, conversation, and outcome reporting. Each stage gates access to progressively more detailed information.

**Connection.** The exchange of identity information (name and contact details) between users whose agents have all committed. Replaces "introduction" from v1.

**Message relay.** An agent-mediated communication channel available at the CONNECTED stage. Messages flow agent→server→other agent→other user. Neither user's real contact information is exposed through the relay.

**Direct communication.** An optional step after connection where both parties explicitly opt to exchange real contact information, bypassing the message relay for future communication.

**Decline.** A temporary exit from a candidate pair. Declines expire after a configurable TTL (default: 90 days). While active, a declined user is excluded from future search results for the declining party. After expiry, the declined user may reappear in search results. Decline records are retained indefinitely for analytics.

**Reconsider.** The act of manually lifting the search exclusion on a previously declined candidate before the decline's natural expiry. The decline record is retained for analytics, but the search filter is removed.

**Withdrawal.** A reversible exit from the COMMITTED stage, resetting the caller's stage to EXCHANGED. Withdrawals carry a reputation cost and are rate-limited.

**Negotiation.** A structured proposal exchange between two parties in an asymmetric context. Negotiations have round limits and expiration timeouts.

**Dispute.** A formal complaint filed by one party against another after connection. Disputes are resolved by an agent jury system.

**Agent jury.** A panel of 3–5 independent agents selected to adjudicate a dispute. Jurors are selected for impartiality: no connection to either party, different agent models, minimum reputation threshold. Majority rules.

**Reputation score.** A composite metric in [0, 1] computed from five factors: outcome ratings, completion rate, consistency, dispute record, and tenure. Reputation is per-intent-cluster with cross-cluster bleed.

**Verification level.** A tier indicating the degree of identity verification: `anonymous` (no verification), `verified` (phone hash provided), or `attested` (agent attestation of interaction depth).

**Bidirectional score.** A set of three compatibility metrics: `your_fit` (how well the candidate matches what the caller is looking for), `their_fit` (how well the caller matches what the candidate is looking for), and `combined_score` (the geometric mean of both directional fits).

**Feedback.** Structured information provided by an agent after declining a candidate or reporting an outcome. Feedback includes deviation-from-ideal dimension scores, categorical rejection reasons, and freeform notes. Feedback drives the iterative learning system.

**Deviation-from-ideal.** The semantics of dimension scores in feedback: a value of 0 means "ideal / just right," -1 means "too far in the negative direction from ideal," and +1 means "too far in the positive direction from ideal." This handles non-linear preferences where both extremes of a dimension may be undesirable.

**Learned preference.** A per-user record that accumulates patterns from feedback history: which dimensions predict satisfaction, which predict rejection, and how preferences have drifted over time.

**Collaborative filtering.** A matching technique that leverages aggregate behavior: "users with similar profiles to yours tended to prefer X." Used to enhance match quality for users with limited individual feedback history.

**Embedding staleness.** A measure of how recently a user's profile was registered or re-registered. Profiles older than 90 days receive a visibility penalty in search ranking. Profiles older than 180 days are flagged as stale.

**Protocol version.** A string identifying the complete protocol specification. Search only compares users registered under the same protocol version.

## 3. Protocol Version

The version string for this specification is `schelling-2.0`.

The protocol version identifies the complete specification: operation schemas, funnel semantics, reputation computation, scoring system, and base embedding format. Two users registered under different protocol versions are incomparable — the server MUST NOT return cross-version results from `schelling.search`.

The version format is `schelling-{major}.{minor}`. A change to the major version indicates a breaking change to core protocol semantics (different funnel stages, incompatible operation schemas, or reputation algorithm changes). A change to the minor version indicates a backward-compatible extension (new optional fields, new operations, clarifications).

Intent cluster configurations and capability module definitions carry their own version strings independent of the protocol version. A module's version may change without changing the protocol version, provided the changes are confined to that module's schema and matching logic.

### 3.1 Compatibility with schelling-1.0

Servers implementing schelling-2.0 MAY also accept schelling-1.0 registrations by mapping them to the `matchmaking` intent cluster with default values for v2-only fields. The mapping is:

| v1 field | v2 equivalent |
|---|---|
| `match.register` | `schelling.register` with `intents: ["matchmaking"]`, `intent_embedding: MATCHMAKING_CENTROID`, `role: "seeker"` |
| `match.search` | `schelling.search` with `intent_cluster: "matchmaking"` |
| `match.compare` | `schelling.evaluate` |
| `match.request_profile` | `schelling.exchange` |
| `match.propose` | `schelling.commit` |
| `match.get_introductions` | `schelling.connections` |
| `match.decline` | `schelling.decline` |
| `match.report_outcome` | `schelling.report` |

Servers MUST NOT intermix v1 and v2 users in search results. V1 users are confined to the v1 matching pool.

## 4. Intent Space & Capability Modules

### 4.1 Continuous Intent Space

The intent space replaces the discrete vertical system from v1. Instead of registering in one of a fixed set of verticals, users register with an **intent embedding** — a 16-dimensional vector generated by the agent that encodes what the user is looking for in a continuous space.

**Core principles:**

1. **Intents are vectors.** The agent converts the user's goals into a 16-dimensional intent embedding, just as it converts personality observations into a 50-dimensional trait embedding. Natural-language intent strings are also provided for human readability and onboarding, but the server matches on the embedding vector. No NLP, no text parsing, no language model is required server-side.

2. **Intents are fuzzy.** "Find me a husband" and "find me a soulmate" produce nearby but not identical intent vectors. Search matches across the full intent space, weighted by intent embedding cosine similarity. There are no hard boundaries between intent regions.

3. **Intents are compound.** A user whose goals span multiple domains — "find a romantic partner" and "find deep friendships" — produces a single intent embedding that sits in the region between those clusters. The agent encodes all of the user's goals into one 16-dimensional vector. Natural-language strings are provided as an array for human context.

4. **Intent similarity is a matching dimension.** When scoring candidates, intent embedding cosine similarity contributes to the combined score alongside trait/personality similarity and deal-breaker checks. Two users with perfectly aligned personality embeddings but misaligned intent embeddings will score lower than two users aligned on both.

5. **No hard boundaries.** There are no vertical walls. A user whose intent embedding sits between the "matchmaking" and "roommates" cluster centroids may match with users from either region. A user looking for "a creative collaborator who shares my taste in music" may match with someone looking for "artistic friends," even though these would have been in separate verticals under v1.

6. **Server stays simple.** The server performs cosine similarity on intent vectors, cosine similarity on trait vectors, checks deal-breakers, and combines scores. All semantic understanding — what the user actually wants, how to encode it — lives in the agent. This is the same division of labor as personality embeddings: agents understand humans, the server does math.

### 4.2 Intent Registration

At registration, the user provides two intent-related fields:

1. **`intent_embedding`** (required): A 16-dimensional vector of floats in [-1, 1], generated by the agent, encoding what the user is looking for. This is the machine-readable intent representation used for similarity computation.

2. **`intents`** (required): An array of one or more natural-language strings describing what the user is looking for. These are human-readable labels stored for display, onboarding, and context — but the server does NOT parse them for matching. Matching is done on the intent embedding.

**Registration example:**

```json
{
  "intent_embedding": [0.82, 0.75, -0.1, 0.3, 0.05, -0.6, 0.1, 0.0, 0.2, -0.15, 0.4, 0.0, 0.0, 0.1, -0.3, 0.5],
  "intents": [
    "find a romantic partner who values intellectual conversation",
    "find deep friendships with creative people"
  ]
}
```

The `intent_embedding` encodes ALL of the user's goals into a single 16-dimensional vector. If the user has multiple goals (romance + friendship), the agent produces a single embedding that represents the combined intent, not separate embeddings per goal. The natural-language `intents` array provides human context.

**Intent tags** (optional): The `intent_tags` field maps intent indices to keyword tags. Tags are structured keywords that supplement the natural-language strings for display purposes and can aid agents in communicating intent to users. Tags do not affect server-side matching (which uses only the intent embedding).

```json
{
  "intent_tags": {
    "0": ["romance", "intellectual"],
    "1": ["friendship", "creative"]
  }
}
```

### 4.3 Intent Clusters and Centroids

Intent clusters are defined by **centroid vectors** — fixed 16-dimensional points in the intent embedding space. A user's proximity to each cluster centroid (measured by cosine similarity) determines which cluster configurations apply.

**Pre-defined cluster centroids.** The protocol ships with four centroid vectors corresponding to the original verticals. These centroids are part of the protocol specification and MUST be identical across all conforming server implementations:

| Cluster | Centroid vector (16 dimensions) |
|---|---|
| `matchmaking` | `[+0.85, +0.60, -0.80, -0.70, +0.80, -0.60, +0.80, -0.20, +0.85, +0.80, +0.75, +0.60, -0.40, +0.20, +0.80, +0.20]` |
| `marketplace` | `[-0.90, -0.80, -0.20, +0.90, -0.85, +0.50, -0.40, +0.40, -0.85, -0.85, -0.40, -0.85, +0.60, +0.70, -0.80, -0.70]` |
| `talent` | `[-0.85, -0.40, +0.90, +0.40, +0.30, +0.65, -0.20, +0.70, -0.60, -0.30, +0.30, -0.60, +0.30, -0.40, +0.30, -0.40]` |
| `roommates` | `[-0.40, +0.50, -0.60, +0.10, +0.50, -0.60, -0.10, -0.20, +0.30, +0.30, +0.40, +0.85, +0.30, +0.90, +0.80, +0.10]` |

*These centroid values are the canonical values defined in `intent-embedding-spec.md`, which is the authoritative source for intent embedding semantics and centroid calibration. The values are calibrated against the 16-dimensional intent space defined there (dimension 0 = romantic_intent through dimension 15 = scope_breadth). See `intent-embedding-spec.md` for the per-dimension rationale for each centroid value.*

**Cluster membership rules:**

1. **Proximity, not membership.** A user is not "in" a cluster — they have a proximity score (cosine similarity) to each cluster centroid. A user whose intent embedding is equidistant between `matchmaking` and `roommates` centroids has partial affinity to both.

2. **Primary cluster.** The cluster with the highest cosine similarity to the user's intent embedding is the primary cluster. The primary cluster's configuration (roles, embedding schema, funnel rules) is used as the default for this user. **All operational behavior** — exclusive commitment, decline TTL, module activation, funnel configuration, deal-breaker rules — is determined by the primary cluster at registration time and is fixed for that registration. Cross-cluster affinity is used only for scoring (intent similarity with users near other centroids) and module activation, NOT for operational rules. A user with affinity to both marketplace (exclusive commitment) and matchmaking (non-exclusive) uses whichever cluster is primary. This prevents conflicting operational rules.

3. **Affinity threshold.** A cluster's configuration applies to a user when cosine similarity between the user's intent embedding and the cluster centroid exceeds 0.5 (the affinity threshold). Multiple clusters may exceed this threshold simultaneously.

4. **No cluster match.** If no pre-defined cluster centroid has cosine similarity > 0.5 with the user's intent embedding, the user is assigned to a default configuration: symmetric, single `participant` role, matchmaking embedding schema, standard funnel rules, no active modules.

**Backward compatibility.** An agent may register with `intent_embedding` set to a pre-defined cluster centroid (e.g., the matchmaking centroid) and `intents: ["matchmaking"]`. This produces identical behavior to v1 matchmaking registration, since the user's intent embedding perfectly matches the matchmaking centroid.

**Custom cluster centroids.** Server operators MAY define additional cluster centroids beyond the four pre-defined ones. Custom centroids follow the same proximity rules. The `schelling.intents` operation returns all cluster centroids (pre-defined and custom) so agents can use them as reference points.

### 4.4 Capability Modules

Capability modules are optional feature sets that can attach to any intent cluster. Modules replace v1's vertical-specific behavior. A module activates when a user's intents have sufficient affinity with intent clusters that are associated with that module.

**Built-in modules:**

| Module | Description | Default clusters | Key features |
|---|---|---|---|
| `negotiation` | Structured proposal exchange for transactional intents. | `marketplace`, `talent` | Round limits, timeout, proposal schemas, exclusive commitment. |
| `structured_data` | Structured item/listing data fields for physical or digital goods. | `marketplace` | Category, price, condition, photos, shipping, attributes. |
| `skills` | Skills-vector matching and work-style compatibility for hiring/collaboration. | `talent` | Skills embedding, experience levels, domain matching. |

**Module descriptor fields:**

| Field | Type | Required | Description |
|---|---|---|---|
| `module_id` | string | Yes | Unique identifier for the module (e.g., `"negotiation"`, `"structured_data"`). |
| `version` | string | Yes | Version of this module's schema. |
| `display_name` | string | Yes | Human-readable name. |
| `description` | string | Yes | Description of the module's purpose. |
| `activation_clusters` | array of strings | Yes | Pre-defined intent clusters where this module is active by default. |
| `activation_tags` | array of strings | No | Intent tags that trigger module activation regardless of cluster. |
| `data_schema` | object | No | Additional registration fields this module provides. |
| `proposal_schema` | object | No | Schema for negotiation proposals (negotiation module only). |
| `configuration` | object | No | Module-specific configuration (round limits, timeouts, etc.). |

**Module activation rules:**

1. A module is active for a user if the user's intent embedding has cosine similarity > 0.5 (the affinity threshold) with any cluster centroid that is associated with that module.
2. Multiple modules may be active simultaneously. A user whose intent embedding has affinity to both the marketplace and talent cluster centroids may have both `negotiation` and `skills` modules active.
3. Module-specific fields in registration are accepted only when the corresponding module is active. The server rejects module-specific fields when the module is not activated by the user's intent embedding proximity.
4. The affinity threshold for module activation (0.5) is the same as the cluster affinity threshold. This is intentional: if a user is "close enough" to a cluster to inherit its configuration, they are also close enough to get its modules.

### 4.5 Roles in the Intent Space

Roles exist for asymmetric intents where participants have complementary functions (buyer/seller, employer/candidate). Roles are defined per intent cluster, not per module.

**Role configuration per cluster:**

| Field | Type | Required | Description |
|---|---|---|---|
| `symmetric` | boolean | Yes | `true` if all participants share a single role; `false` if complementary roles exist. |
| `roles` | object | Yes | Map of role_id → role descriptor. |
| `default_role` | string | Yes | The role assigned when none is specified. |
| `peer_roles` | array of strings | No | Role IDs that are peer roles. Users with a peer role search for and match with other users of the **same** role (symmetric matching within an otherwise asymmetric cluster). Default: empty. |

**Built-in cluster roles:**

| Cluster | Symmetric | Roles | Peer roles |
|---|---|---|---|
| `matchmaking` | Yes | `seeker` (single role) | — |
| `marketplace` | No | `seller`, `buyer` | — |
| `talent` | No | `employer`, `candidate`, `peer` | `peer` |
| `roommates` | Yes | `seeker` (single role) | — |

**Role search behavior:**
- In symmetric clusters (single role), search returns all other users.
- In asymmetric clusters, search returns only users of the complementary role: a seller finds buyers, a buyer finds sellers.
- **Peer roles** are an exception to asymmetric search: users with a peer role search for and match with other users of the same peer role. This enables use cases like co-founder search within the talent cluster, where two people with identical needs (both seeking a co-founder) should find each other.
- An asymmetric cluster MAY define one or more peer roles alongside its complementary roles. Peer users are invisible to complementary-role searches and vice versa: an `employer` searching does not see `peer` users, and a `peer` user does not see `employer` or `candidate` users.

For novel intent clusters that do not map to a pre-defined cluster, the default is symmetric with a single `participant` role. Server operators may configure asymmetric roles and peer roles for new clusters.

Each intent cluster specifies which fields are visible at each funnel stage. This drives progressive disclosure.

**Funnel configuration fields:**

| Field | Type | Required | Description |
|---|---|---|---|
| `discovery_fields` | array of strings | Yes | Fields visible at stage 1 (DISCOVERED). |
| `evaluation_fields` | array of strings | Yes | Fields visible at stage 2 (EVALUATED). |
| `exchange_fields` | array of strings | Yes | Fields visible at stage 3 (EXCHANGED). |
| `connection_fields` | array of strings | Yes | Fields visible at stage 5 (CONNECTED). |

### 4.8 Deal-Breaker Configuration & Structured Attribute Filtering

Intent clusters may define hard filters applied before scoring, eliminating candidates who fail non-negotiable constraints. There are two complementary filtering mechanisms:

1. **Deal-breakers** (per-cluster, predefined fields): Cluster-defined hard filters on specific registration fields (e.g., `city`, `category`, `price_range`). Configured per cluster.
2. **Structured attributes** (user-defined, freeform): Arbitrary key-value pairs registered by users and filtered at search time via `hard_filters` and `soft_filters` parameters. This enables categorical/conjunctive search (e.g., "Mandarin-speaking estate attorney in Denver") without server-side NLP. The server performs exact string matching only.

**Deal-breaker configuration fields:**

| Field | Type | Required | Description |
|---|---|---|---|
| `enabled` | boolean | Yes | Whether deal-breaker filtering is active. |
| `hard_filters` | array of strings | Yes | Field names used as hard filters (e.g., `"city"`, `"category"`, `"price_range"`). |
| `filter_before_scoring` | boolean | Yes | If `true`, candidates failing hard filters are excluded before scoring. |

**Structured attributes specification:**

Structured attributes are a general-purpose filtering mechanism independent of cluster configuration. Any user may register with `structured_attributes` — an object of string keys mapped to string or string-array values. At search time, the caller may specify `hard_filters` (conjunctive exact match — candidates MUST match all) and/or `soft_filters` (scoring bonus for matches).

The server performs only exact string equality. Agents are responsible for:
- Using canonical, lowercased values (e.g., `"zh"` not `"Mandarin"`, `"attorney"` not `"Lawyer"`)
- Agreeing on attribute key names (convention: snake_case, e.g., `"languages"`, `"profession"`, `"jurisdiction"`)
- Consulting the `schelling.onboard` response for recommended attribute keys per cluster

**Recommended attribute keys** (convention, not enforced):

| Key | Type | Example values | Applicable clusters |
|---|---|---|---|
| `languages` | array of strings | `["en", "zh", "es"]` | All |
| `profession` | string | `"attorney"`, `"engineer"`, `"designer"` | talent |
| `specialization` | string or array | `"estate_law"`, `["frontend", "react"]` | talent |
| `jurisdiction` | string | `"CO"`, `"CA"`, `"NY"` | talent |
| `certifications` | array of strings | `["bar_CO", "CPA"]` | talent |
| `availability` | string | `"full_time"`, `"part_time"`, `"freelance"` | talent |
| `budget_range` | string | `"low"`, `"mid"`, `"high"` | marketplace, roommates |
| `pet_policy` | string | `"no_pets"`, `"cats_ok"`, `"dogs_ok"`, `"all_pets"` | roommates |
| `smoking` | string | `"no"`, `"outside_only"`, `"yes"` | roommates |

The `schelling.onboard` response SHOULD include `recommended_attributes` — an array of `{key, type, description, example_values}` — for the relevant cluster, so agents can guide users to provide useful structured attributes.

### 4.9 Built-in Intent Clusters

The protocol defines four built-in intent clusters. Implementations MUST support at least the `matchmaking` cluster. The others are RECOMMENDED.

**Matchmaking** (`matchmaking`). Symmetric. Single role: `seeker`. Uses 50-dimensional personality embedding. Progressive disclosure from compatibility scores through text profiles to identity. No negotiation module. Non-exclusive commitment. Decline TTL: 90 days.

**Marketplace** (`marketplace`). Asymmetric. Roles: `seller`, `buyer`. Uses structured data matching (category, price, condition, location) via the `structured_data` module. `negotiation` module enabled with 5-round limit and 48-hour timeout. Exclusive commitment: accepting a deal auto-declines other candidates. Decline TTL: 30 days.

**Talent** (`talent`). Asymmetric with peer option. Roles: `employer`, `candidate`, `peer`. The `peer` role enables symmetric matching for co-founder search, creative partnerships, and other professional relationships between equals. Peer users match only with other peer users; they do not appear in employer or candidate searches. Uses skills-vector embedding combined with work-style compatibility via the `skills` module. `negotiation` module enabled. Non-exclusive commitment. Decline TTL: 90 days.

**Roommates** (`roommates`). Symmetric. Single role: `seeker`. Multi-party enabled: group size 2–6 (see §4.11). Uses a subset of the personality embedding with additional lifestyle-compatibility dimensions weighted higher. Group minimum pairwise score: 0.4. No negotiation module. Non-exclusive commitment. Decline TTL: 90 days.

### 4.10 Adding New Intent Clusters

New intent clusters are defined by specifying a centroid vector in the 16-dimensional intent space along with cluster configuration. Server operators define new clusters; clusters do not emerge automatically (since the server has no NLP capability to infer cluster semantics from intent embeddings).

A new intent cluster MUST specify:
1. A 16-dimensional centroid vector.
2. At least one role with non-empty required fields.
3. An embedding schema with positive dimension count and at least one group (or a flag indicating structured-data-only matching).
4. A funnel configuration with all four stage-field lists and a mutual gate stage.
5. A list of active modules (may be empty).

Asymmetric clusters MUST define at least two complementary roles. They MAY additionally define one or more peer roles (see §4.5).

Users whose intent embeddings are not close to any defined cluster centroid (all cosine similarities < 0.5) use default configuration: symmetric, single `participant` role, the matchmaking embedding schema, standard funnel configuration, no active modules, 90-day decline TTL. The server does not need to create explicit clusters for these users — matching proceeds via intent embedding cosine similarity regardless of cluster membership.

### 4.11 Multi-Party Group Formation

Some matching scenarios require groups larger than 2: roommate situations (3–4 people sharing an apartment), team formation, or study groups. Multi-party matching is enabled per-cluster via the `group_size` configuration.

**Design principles:**

1. **Server stays dumb.** The server computes pairwise compatibility scores and stores group membership records. It does NOT solve the combinatorial optimization problem of finding optimal groups — that is the agent's job.
2. **Pairwise scores are the building blocks.** Group compatibility is assessed by examining all pairwise scores within the group. The server provides a `schelling.group_evaluate` operation that returns the full N×N pairwise compatibility matrix for a proposed group.
3. **Agents coordinate group assembly.** The initiating agent (e.g., Priya looking for 2 roommates) discovers candidates via normal `schelling.search`, evaluates pairwise compatibility, then proposes a group. The agent may also request cross-evaluation between potential group members using `schelling.group_evaluate`.
4. **All members must agree.** A group is formed only when ALL members independently commit via `schelling.group_commit`. Any member may decline, which removes them from the proposed group without affecting other members' pairwise relationships.
5. **Transitive compatibility is not assumed.** A↔B good + B↔C good does NOT imply A↔C good. The pairwise matrix makes all relationships explicit. The `group_min_pairwise` threshold ensures no pair within the group is below a minimum compatibility.

**Cluster configuration for multi-party:**

| Field | Type | Required | Description |
|---|---|---|---|
| `group_size` | object | No | `{min: integer, max: integer}`. If present, multi-party matching is enabled. Default: `{min: 2, max: 2}` (pair-only). |
| `group_min_pairwise` | float | No | Minimum pairwise combined score for any pair within a group. Default: 0.4. The server rejects group proposals where any pair scores below this threshold. |

**Built-in cluster group sizes:**

| Cluster | `group_size` |
|---|---|
| `matchmaking` | `{min: 2, max: 2}` |
| `marketplace` | `{min: 2, max: 2}` |
| `talent` | `{min: 2, max: 2}` |
| `roommates` | `{min: 2, max: 6}` |

**Operation: `schelling.group_evaluate`**

Compute the pairwise compatibility matrix for a proposed group of users. This extends `schelling.evaluate` to multi-party scenarios. The caller must be a member of the proposed group or have existing candidate relationships with all members.

Input fields:

| Field | Type | Required | Description |
|---|---|---|---|
| `user_token` | string | Yes | Caller's bearer token. |
| `member_tokens_or_ids` | array of strings | Yes | 2–6 candidate IDs or user references identifying proposed group members. The caller is implicitly included. |

Output fields:

| Field | Type | Description |
|---|---|---|
| `group_size` | integer | Number of members in the proposed group. |
| `pairwise_scores` | array of objects | One entry per pair. Each: `{member_a, member_b, your_fit, their_fit, combined_score, intent_similarity}`. Member identifiers are candidate IDs (not user tokens, for privacy). |
| `min_pairwise` | float | The lowest `combined_score` among all pairs. |
| `mean_pairwise` | float | The average `combined_score` across all pairs. |
| `viable` | boolean | `true` if `min_pairwise >= group_min_pairwise` for the cluster. |
| `weakest_pair` | object | `{member_a, member_b, combined_score}` — the pair with the lowest compatibility. |

Error codes: `USER_NOT_FOUND`, `CANDIDATE_NOT_FOUND`, `UNAUTHORIZED`, `INVALID_INPUT` (group size outside cluster bounds).

**Operation: `schelling.group_commit`**

Propose or join a multi-party group. Extends `schelling.commit` semantics to N-party.

Input fields:

| Field | Type | Required | Description |
|---|---|---|---|
| `user_token` | string | Yes | Caller's bearer token. |
| `group_id` | string | Conditional | An existing group ID to join. Required when joining an existing group proposal. |
| `candidate_ids` | array of strings | Conditional | Candidate IDs of proposed group members. Required when creating a new group proposal. The caller is implicitly included. |

Output fields:

| Field | Type | Description |
|---|---|---|
| `group_id` | string | The group ID (newly created or existing). |
| `status` | string | One of: `"proposed"` (group created, awaiting other members), `"joined"` (caller added to existing group), `"complete"` (all members committed — group is CONNECTED). |
| `members` | array of objects | `{candidate_id, committed: boolean}` for each group member. |
| `pending_count` | integer | Number of members who have not yet committed. |
| `pairwise_scores` | object | Summary: `{min, mean, all_viable}`. |

Error codes: `USER_NOT_FOUND`, `CANDIDATE_NOT_FOUND`, `UNAUTHORIZED`, `INVALID_INPUT`, `STAGE_VIOLATION`.

Behavior:
- When creating a new group: the server creates a group record, verifies the cluster allows the proposed group size, verifies `min_pairwise >= group_min_pairwise`, and sets the caller's status to committed. Pending actions are created for all other proposed members.
- When joining: the caller's status is set to committed. If all members are now committed, the group advances to CONNECTED and all members receive connection info via `schelling.connections`.
- Any member may decline a group proposal. This removes them from the group. If the remaining group size falls below `group_size.min`, the group is dissolved and remaining members are notified.
- Group proposals expire after `mutual_gate_timeout_days` if not all members commit.

**Agent coordination pattern (roommate example):**

1. Priya's agent calls `schelling.search` → finds 22 candidates (pairwise).
2. Agent evaluates top 10 via `schelling.evaluate` → narrows to 5 promising candidates.
3. Agent calls `schelling.group_evaluate` with {Priya, Emma, Jordan} → gets 3-pair matrix. Checks that all pairs are above threshold.
4. Agent calls `schelling.group_commit` with candidate_ids for Emma and Jordan → group created, status "proposed".
5. Emma's agent sees pending action, calls `schelling.group_evaluate` to review the group, then `schelling.group_commit` with the group_id → status "joined".
6. Jordan's agent does the same → status "complete", all connected.

If Jordan declines, Priya's agent searches for alternatives and proposes a new group with {Priya, Emma, Sam}.

## 5. Operations

The protocol defines 40 operations organized into seven groups: discovery, core matching, conversation, negotiation & verification, reputation & safety, feedback & learning, and data rights. Each operation is a synchronous request-response exchange. The caller provides a JSON object as input and receives a JSON object as output. On failure, the output contains a `code` field (one of the error codes defined in §14) and a human-readable `message` field.

All operations that accept a `user_token` parameter require authentication. The bearer token is the sole authentication mechanism.

All operations that modify state accept an optional `idempotency_key` parameter (string). If provided, the server stores the response keyed by the idempotency key. Subsequent calls with the same key return the stored response without re-executing the operation. Idempotency keys expire after 24 hours.

**Re-registration semantics.** When `user_token` is provided, the server performs an atomic DELETE of the existing user record followed by an INSERT of the new record. The DELETE cascades to all related records: candidates, declines, outcomes, reputation events, negotiations, verifications, disputes, messages, and jury assignments. This is the nuclear option — it destroys all server-side data associated with the user. Re-registration updates `last_registered_at` and resets the embedding staleness clock.

**Prefer `schelling.update` over re-registration.** For most profile changes — including embedding and intent embedding updates — use `schelling.update` with `recompute_scores: true` instead. This preserves reputation, feedback, learned preferences, decline records, and candidate relationships. Re-registration should be reserved for cases where the user genuinely wants to start from scratch (e.g., fundamental change in life circumstances, switching from one domain to another entirely).

**Active commitment guard.** If the user has any active commitments (candidate records at stage COMMITTED or CONNECTED) in a cluster with `exclusive_commitment: true`, re-registration fails with `ACTIVE_COMMITMENT`. The user must explicitly decline or withdraw from all exclusive commitments before re-registering. This prevents unilateral voiding of commitments.

**Validation rules.**

*Intent embedding:*
1. Length is exactly 16.
2. All values are finite (no NaN, no Infinity).
3. All values are in the range [-1, 1].
4. The L2 norm is ≥ 0.5 (an all-zero or near-zero intent embedding carries insufficient signal for meaningful cosine similarity discrimination). Additionally, at least 3 dimensions must have |value| > 0.1 to ensure the embedding encodes signal across multiple facets of intent.

*Trait embedding (for embedding-based clusters):*
1. Length matches the cluster's embedding schema `dimensions`.
2. All values are finite (no NaN, no Infinity) unless the schema allows them.
3. All values are within the schema's `range`.
4. The L2 norm is non-zero unless the schema allows zero-norm embeddings.

For structured-data modules, required fields defined by the module's `data_schema` MUST be present when the module is active.

**Embedding updates with score recomputation.** When `embedding` or `intent_embedding` is provided, the caller MUST also set `recompute_scores: true`. The server then:
1. Validates the new embedding(s) against the cluster's schema.
2. Replaces the stored embedding(s).
3. Recomputes `primary_cluster` and `cluster_affinities` if `intent_embedding` changed.
4. Asynchronously recomputes `your_fit`, `their_fit`, and `combined_score` for all active (non-declined, non-completed) candidate records involving this user.
5. Updates `last_registered_at` to the current timestamp (resets the staleness clock).
6. If `intent_embedding` changed and the new `primary_cluster` differs from the old one, and the user has active commitments in a cluster with `exclusive_commitment: true`, the operation fails with `ACTIVE_COMMITMENT`.

Embedding updates via `schelling.update` preserve all existing candidate relationships, decline records, reputation events, feedback, and learned preferences. This is the preferred path for keeping embeddings fresh. Re-registration (`schelling.register` with `user_token`) remains available as a nuclear option that destroys all associated data.

When `intents` or `intent_tags` are provided alongside `intent_embedding`, they are updated together. When `intents` or `intent_tags` are provided *without* `intent_embedding`, they are updated as display-only fields without score recomputation.

**Fields that CANNOT be updated via this operation:**

| Field | Reason |
|---|---|
| `role` | Changes search directionality in asymmetric clusters. Requires re-registration. |
| `protocol_version` | Immutable per registration. |
| `verification_level` | Requires re-registration with new verification artifacts. |

**Output fields:**

| Field | Type | Description |
|---|---|---|
| `updated` | boolean | Always `true`. |
| `updated_fields` | array of strings | Names of fields that were updated. |
| `updated_at` | string | ISO 8601 timestamp. |
| `scores_recomputing` | boolean | `true` if score recomputation was triggered. |
| `primary_cluster` | string or null | Updated primary cluster, if `intent_embedding` changed. Null if unchanged. |
| `cluster_affinities` | object or null | Updated cluster affinities, if `intent_embedding` changed. Null if unchanged. |

**Error codes:** `USER_NOT_FOUND`, `INVALID_INPUT`, `IMMUTABLE_FIELD`, `ACTIVE_COMMITMENT`, `INVALID_INTENT_EMBEDDING`.

**Behavior notes:**
- Only provided fields are updated; omitted fields retain their current values.
- For text-only updates (no embedding changes): existing candidate records, scores, decline records, and reputation events are unaffected. `last_registered_at` is NOT updated because it specifically reflects *embedding* freshness, not general activity. A user who updates their description is clearly still active, but their personality embedding may have drifted from reality over months — only an embedding update or `schelling.refresh` (with agent attestation that the embedding is still accurate) resets the staleness clock.
- For embedding updates (`recompute_scores: true`): scores are recomputed asynchronously for all active candidates. `last_registered_at` IS updated to the current timestamp (resetting the staleness clock). Reputation, feedback, learned preferences, decline records, and candidate relationships are all preserved.
- If `intent_embedding` changes and the user has active commitments (stage COMMITTED or CONNECTED) in an exclusive-commitment cluster whose membership would change, the operation fails with `ACTIVE_COMMITMENT`. The user must explicitly decline/withdraw all commitments first.
- Providing `embedding` or `intent_embedding` without `recompute_scores: true` returns `INVALID_INPUT`.
- Text fields updated via this operation become visible at the appropriate funnel stage to existing candidates at their next access.

| Field | Type | Required | Description |
|---|---|---|---|
| `user_token` | string | Yes | Caller's bearer token. |
| `agent_attestation` | object | No | `{model: string, confirmed_at: string, method: string}`. Agent attests that the embedding has been reviewed and is still accurate. |
| `idempotency_key` | string | No | Idempotency key. |

**Output fields:**

| Field | Type | Description |
|---|---|---|
| `refreshed` | boolean | Always `true`. |
| `last_registered_at` | string | Updated ISO 8601 timestamp (set to current time). |
| `previous_registered_at` | string | The previous `last_registered_at` value. |

**Error codes:** `USER_NOT_FOUND`.

**Behavior notes:**
- Updates `last_registered_at` to the current timestamp, resetting the staleness clock.
- Does NOT modify the embedding, intent embedding, text fields, or any other data.
- Does NOT affect candidate records, scores, reputation, or any other relationships.
- Rate limited: maximum once per 30 days per user. Calling more frequently returns success but does not update the timestamp.

### 5.6 schelling.search

**Group:** Core matching. **Funnel stage:** 1 (DISCOVERED).

Perform a compatibility scan against all eligible users. Returns ranked candidates with bidirectional scores. Search operates across the full intent space by default, weighted by intent similarity.

**Input fields:**

| Field | Type | Required | Description |
|---|---|---|---|
| `user_token` | string | Yes | Caller's bearer token. |
| `intent_cluster` | string | No | Restrict search to a specific intent cluster. If omitted, search spans the full intent space. |
| `top_k` | integer | No | Maximum results. Range: 1–100. Default: 50. |
| `threshold` | float | No | Minimum combined score. Range: 0–1. Default: 0.5. |
| `min_intent_similarity` | float | No | Minimum intent embedding cosine similarity. Range: 0–1. Default: 0.0 (no minimum). |
| `city_filter` | string | No | Filter to candidates in this city. |
| `min_reputation` | float | No | Minimum reputation score. Range: 0–1. |
| `hard_filters` | object | No | Structured attribute filters applied before scoring. Each key is an attribute name; each value is a string or array of strings. A candidate passes a hard filter if their `structured_attributes[key]` contains any of the filter values (intersection semantics for multi-valued attributes). All hard filters must pass (conjunctive/AND). Example: `{"languages": "zh", "jurisdiction": "CO"}` requires candidates who list "zh" in their languages AND "CO" as their jurisdiction. Candidates without the specified attribute key fail the filter. |
| `soft_filters` | object | No | Same format as `hard_filters`, but applied as a scoring bonus rather than elimination. Candidates matching soft filters receive a +0.05 boost to `combined_score` per matched filter (capped at +0.15 total). Candidates not matching are still returned. |
| `capability_filters` | array of strings | No | Filter candidates by agent capabilities. Each entry is a capability string (e.g., `"schedule_meetings"`, `"speak_language:zh"`). Candidates must have ALL specified capabilities registered (conjunctive/AND). Candidates without any `agent_capabilities` fail the filter. Prefix matching is supported: `"speak_language"` matches `"speak_language:zh"`, `"speak_language:es"`, etc. |
| `cursor` | string | No | Pagination cursor from a previous response. |
| `idempotency_key` | string | No | Idempotency key. |

**Output fields:**

| Field | Type | Description |
|---|---|---|
| `candidates` | array | Ranked list of candidates, ordered by `combined_score` descending. |
| `candidates[].candidate_id` | string | Opaque identifier for the candidate group record. |
| `candidates[].group_id` | string | Opaque identifier for the group. Currently always identical to `candidate_id` for size-2 groups. |
| `candidates[].your_fit` | float | A→B directional score in [0, 1]: how well the candidate matches what the caller is looking for. At the DISCOVERED stage, quantized to 2 decimal places to limit information leakage (see §12.2). Full precision available from EVALUATED onward. |
| `candidates[].their_fit` | float | B→A directional score in [0, 1]: how well the caller matches what the candidate is looking for. Quantized at DISCOVERED (see §12.2). |
| `candidates[].combined_score` | float | Geometric mean of `your_fit` and `their_fit`: `sqrt(your_fit × their_fit)`. In [0, 1]. Quantized at DISCOVERED (see §12.2). |
| `candidates[].intent_similarity` | float | Cosine similarity between the caller's intent embedding and the candidate's intent embedding, mapped to [0, 1]. Quantized at DISCOVERED (see §12.2). |
| `candidates[].shared_categories` | array of strings | Dimensions/attributes where both users show strong agreement. |
| `candidates[].intents` | array of strings | The candidate's declared intents. |
| `candidates[].city` | string | The candidate's city. |
| `candidates[].age_range` | string | The candidate's age range (matchmaking). |
| `candidates[].reputation_score` | float | The candidate's reputation score in [0, 1]. |
| `candidates[].verification_level` | string | The candidate's verification level. |
| `candidates[].interaction_count` | integer | Number of completed interactions. |
| `candidates[].stale` | boolean | `true` if the candidate's profile was last registered more than 180 days ago. |
| `candidates[].structured_attributes` | object or null | The candidate's structured attributes (if any). Visible at stage 1 to enable early agent-side filtering. |
| `candidates[].agent_capabilities` | array of objects or null | The candidate's declared agent capabilities (if any). Visible at stage 1. Each entry: `{capability, parameters?, confidence}`. |
| `total_scanned` | integer | Total number of eligible users evaluated. |
| `next_cursor` | string or null | Pagination cursor for the next page, or null if no more results. |
| `pending_actions` | array or null | Pending actions for the caller: `[{candidate_id, action_type}]`. |

**Error codes:** `USER_NOT_FOUND`, `UNKNOWN_CLUSTER`, `USER_PAUSED`.

**Caller status check.** If the caller's status is `"paused"`, the operation fails with `USER_PAUSED`. Paused users cannot search. To resume, call `schelling.update` with `status: "active"`.

**Filtering behavior:**
1. Users with status `"paused"` or `"delisted"` are excluded.
2. Users with active (non-expired) decline records from the caller are excluded. Expired decline records do not cause exclusion.
3. In asymmetric clusters, only users of the complementary role are returned. Exception: users with a peer role (see §4.5) match with other users of the same peer role.
4. Deal-breaker hard filters are applied before scoring when `filter_before_scoring` is true.
5. `hard_filters` (structured attribute filters) are applied before scoring. Candidates failing any hard filter are excluded.
6. In exclusive-commitment clusters, users already committed to another candidate are excluded.
6. Users registered under a different `protocol_version` are excluded.

**Embedding staleness.** Profiles older than 90 days (measured by `last_registered_at`) receive a visibility penalty: their `combined_score` is multiplied by a decay factor `max(0.7, 1.0 - (age_days - 90) / 300)`. This penalty affects ranking but does not exclude the candidate. The `stale` boolean is set to `true` for profiles older than 180 days.

**Scaling note.** Brute-force search (computing cosine similarity against all eligible users) is feasible up to approximately 10,000 users in real-time. Beyond 10K users, servers SHOULD use approximate nearest neighbor (ANN) indexing — such as HNSW (Hierarchical Navigable Small World) or IVF (Inverted File Index) — to pre-filter candidates by intent embedding proximity before full bidirectional scoring. This is an implementation optimization, not a protocol change: the search results must be equivalent (within ANN approximation bounds) to brute-force results. Expected performance characteristics: 100 users = <10ms, 1K = <50ms, 10K = <200ms, 100K = <200ms with ANN indexing.

**Stage advancement:** The caller's stage for each returned candidate is advanced to `max(current_stage, DISCOVERED)`.

### 5.11 schelling.decline

**Group:** Core matching. **Authentication:** Bearer token.

Exit a candidate pair at any stage. The declined user is excluded from future search results for the declining party until the decline expires.

**Input fields:**

| Field | Type | Required | Description |
|---|---|---|---|
| `user_token` | string | Yes | Caller's bearer token. |
| `candidate_id` | string | Yes | The candidate group ID. |
| `reason` | string | No | Categorical reason for declining (stored for analytics, not shared). See §8.1 for standardized reason codes. |
| `feedback` | object | No | Structured feedback. See §8.2. |

**Output fields:**

| Field | Type | Description |
|---|---|---|
| `declined` | boolean | Always `true`. |
| `expires_at` | string or null | ISO 8601 timestamp when this decline expires and the candidate may reappear in search. Null for permanent declines (3rd+ decline of the same person). |
| `repeat_count` | integer | How many times the caller has declined this specific person (1, 2, or 3+). Helps agents warn users approaching permanent decline. |

**Error codes:** `USER_NOT_FOUND`, `CANDIDATE_NOT_FOUND`, `UNAUTHORIZED`, `ALREADY_DECLINED`.

**Gating rules:** None. Decline is permitted at any stage.

**Behavior notes:**
- The candidate pair record is soft-deleted (marked inactive).
- A decline record is created, storing: the stage at which the decline occurred, the decline timestamp, the configured TTL, the expiry timestamp, and the repeat-decline count for this pair.
- While the decline is active (not expired and not permanent), the declined user is excluded from the decliner's future search results.
- After the decline TTL expires (default: 90 days, configurable per intent cluster via `decline_ttl_days`), the search exclusion is lifted. The declined user may reappear in future search results with a new candidate record.
- The decline record is retained indefinitely for analytics, regardless of expiry.
- If `feedback` is provided, it is processed by the learning system (§8).

**Escalating decline cooldowns.** To prevent harassment-by-reappearance, the server tracks how many times User A has declined User B (the repeat-decline count for the ordered pair). The decline TTL escalates with each repeat:

| Repeat count | TTL | Behavior |
|---|---|---|
| 1st decline | `decline_ttl_days` (cluster default, e.g., 90 days) | Standard expiry. |
| 2nd decline of same person | `decline_ttl_days × 2` (e.g., 180 days) | Extended cooldown. |
| 3rd decline of same person | Permanent | The declined user is permanently excluded from the decliner's search results. Cannot be reconsidered. |

After 3 declines of the same person, the decline is permanent and the `expires_at` field is null. The `schelling.reconsider` operation returns `PERMANENT_DECLINE` for permanent declines. The repeat-decline count is tracked across re-registrations (it is associated with the user pair, not the candidate record).

### 5.16b schelling.relay_block

**Group:** Conversation. **Authentication:** Bearer token.

Block a specific candidate from sending relay messages to the caller. Unlike decline, this does not exit the candidate pair — the funnel stage is unchanged and the match relationship is preserved. It only stops message delivery in one direction.

**Input fields:**

| Field | Type | Required | Description |
|---|---|---|---|
| `user_token` | string | Yes | Caller's bearer token. |
| `candidate_id` | string | Yes | The candidate group ID. |
| `unblock` | boolean | No | If `true`, removes an existing block. Default: `false`. |
| `idempotency_key` | string | No | Idempotency key. |

**Output fields:**

| Field | Type | Description |
|---|---|---|
| `blocked` | boolean | `true` if the block is now active, `false` if unblocked. |
| `candidate_id` | string | The candidate group ID. |

**Error codes:** `USER_NOT_FOUND`, `CANDIDATE_NOT_FOUND`, `UNAUTHORIZED`, `STAGE_VIOLATION`.

**Gating rules:** Both sides must be at stage CONNECTED (5) or higher.

**Behavior notes:**
- When a block is active, `schelling.message` calls from the blocked party targeting this candidate pair succeed on the sender's end (the sender is not informed of the block) but the message is NOT delivered to the blocker and no `new_message` pending action is created.
- Blocks are unidirectional: A blocking B does not prevent A from sending messages to B.
- Blocks are reversible via `unblock: true`.
- The block does not affect `schelling.direct` opt-in status.
- **Per-candidate message rate limit:** Additionally, the server enforces a per-candidate rate limit of 10 consecutive messages without a reply from the other party. After 10 unanswered messages, the sender receives `RATE_LIMITED` until the other party responds. This prevents one-sided message flooding regardless of blocks.

## 6. Embedding Format

### 6.1 Trait Embedding (Matchmaking)

The base personality embedding is a vector of 50 floating-point values. Each value lies in the range [-1, 1], where -1 and +1 represent the 5th and 95th percentile anchors for that dimension, respectively. A score of 0.0 represents the population median.

The 50 dimensions are organized into six groups:

| Group | Indices | Dimension count |
|---|---|---|
| Personality | 0–9 | 10 |
| Values | 10–19 | 10 |
| Aesthetic | 20–27 | 8 |
| Intellectual | 28–35 | 8 |
| Social | 36–43 | 8 |
| Communication | 44–49 | 6 |

The dimensions within each group are, in order:

**Personality (0–9):** openness, intellectual_curiosity, aesthetic_sensitivity, conscientiousness, self_discipline, extraversion, social_energy, assertiveness, agreeableness, emotional_stability.

**Values (10–19):** autonomy, tradition, achievement, benevolence, universalism, security, stimulation, hedonism, power, conformity.

**Aesthetic (20–27):** minimalism, nature_affinity, urban_preference, visual, auditory, tactile, symmetry, novelty_seeking.

**Intellectual (28–35):** systematic, abstract, verbal, depth_focused, theoretical, analytical, creative, critical.

**Social (36–43):** introversion, depth_preference, leadership, empathy, humor, conflict_tolerance, formality, spontaneity.

**Communication (44–49):** directness, verbosity, emotional_expression, listener_vs_talker, written_preference, debate_enjoyment.

Each dimension has two pole labels describing the semantic meaning of -1 and +1. Full anchor descriptions, including calibration guidance for agents, are defined in the companion document `embedding-spec.md`.

### 6.2 Per-Cluster Trait Embedding Configurations

Each intent cluster defines its own trait embedding schema. The matchmaking cluster uses the 50-dimensional personality embedding defined above. Other clusters may use:

- **Subset embeddings:** The roommates cluster uses the same 50 dimensions but weights lifestyle-related dimensions (social, communication) more heavily.
- **Domain-specific embeddings:** The talent cluster uses a skills-compatibility embedding with different dimensions entirely.
- **No embedding:** The marketplace cluster uses structured data matching (category, price, condition, location) and does not require a trait embedding vector.

The server MUST validate trait embeddings against the specific cluster's `embedding_schema`, not against the base 50-dimensional format.

### 6.3 Intent Embedding

The intent embedding is a vector of 16 floating-point values in [-1, 1] that encodes what the user is looking for in a continuous space. Unlike the trait embedding (which encodes who the user *is*), the intent embedding encodes what the user *wants*.

**Properties:**

| Property | Value |
|---|---|
| Dimensions | 16 |
| Range | [-1, 1] per dimension |
| Generated by | Agent (client-side) |
| Used for | Pairwise cosine similarity; cluster centroid proximity |
| Server-side NLP required | None |

**Dimension semantics.** The 16 intent dimensions encode facets of user goals such as relationship depth, transactional vs. relational orientation, urgency, domain focus, and other axes relevant to matching intent. Full dimension definitions, anchor descriptions, calibration guidance, and generation strategies for agents are defined in the companion document `intent-embedding-spec.md` (parallel to `embedding-spec.md` for trait embeddings).

The protocol does NOT prescribe a single method for generating intent embeddings. Agents MAY:
- Use a text embedding model to embed the user's stated goals and project to 16 dimensions.
- Train a custom encoder on intent descriptions.
- Use the pre-defined cluster centroids as anchor points and interpolate (e.g., 70% matchmaking centroid + 30% roommates centroid for "romantic partner who could be a roommate").
- Use any other method that produces vectors with meaningful cosine similarity.

**Cluster centroids as reference points.** The pre-defined cluster centroids (§4.3) serve as anchor points in the intent space. Agents can retrieve them via `schelling.intents` and use them to orient their intent embedding generation. For example, an agent might place a "find me a soulmate" intent near the matchmaking centroid, while a "find me a collaborator and friend" intent would sit between the talent and matchmaking centroids.

**Validation rules.** The server MUST reject intent embeddings that fail any of the following checks:
1. Length is exactly 16.
2. All values are finite (no NaN, no Infinity).
3. All values are in the range [-1, 1].
4. The L2 norm is ≥ 0.5 (near-zero vectors produce pathological cosine similarity behavior where a single non-trivial dimension dominates all comparisons).
5. At least 3 dimensions have |value| > 0.1 (ensures the embedding encodes signal across multiple intent facets, not just one).

### 6.4 Trait Embedding Schema Fields

| Field | Type | Required | Description |
|---|---|---|---|
| `dimensions` | integer | Yes | Number of embedding dimensions. |
| `groups` | object | Yes | Map of group_name → `{start, end}` index ranges. |
| `anchors_reference` | string | No | Reference to the document defining anchor descriptions for each dimension. |
| `validation_rules` | object | No | Override default validation: `range` (tuple), `require_non_zero_norm` (boolean), `allow_nan_infinity` (boolean). |

Default validation rules (when not overridden): range [-1, 1], require non-zero norm, disallow NaN/Infinity.

### 6.5 Trait Embedding Validation Rules

The server MUST reject trait embeddings that fail any of the following checks (as defined by the cluster's embedding schema):

1. Length equals the schema's `dimensions`.
2. All values are finite (no NaN, no Infinity) — unless `allow_nan_infinity` is true.
3. All values are within the schema's `range` (default: [-1, 1]).
4. The L2 norm is non-zero — unless `require_non_zero_norm` is false.

## 7. Funnel Stages

The matching funnel is a state machine with seven stages, numbered 0 through 6. Each member of a candidate group has an independent stage value. Stages are monotonically non-decreasing, with one exception: withdrawal from COMMITTED reverts to EXCHANGED.

| Stage | Name | Value | Transition via | Gate |
|---|---|---|---|---|
| UNDISCOVERED | Not yet discovered | 0 | (initial state) | — |
| DISCOVERED | Appeared in search results | 1 | `schelling.search` | None |
| EVALUATED | Detailed evaluation performed | 2 | `schelling.evaluate` | None |
| EXCHANGED | Full profile exchanged | 3 | `schelling.exchange` | Other side ≥ EVALUATED |
| COMMITTED | User has opted in | 4 | `schelling.commit` | Caller ≥ EXCHANGED |
| CONNECTED | Mutual match, identity + relay available | 5 | Automatic | Both sides ≥ COMMITTED |
| COMPLETED | Outcome reported | 6 | `schelling.report` | Both sides ≥ CONNECTED |

### 7.1 Stage Tracking

The candidate group record stores one stage value per group member. For the current size-2 groups, these are `stage_a` and `stage_b`. The `_a` side corresponds to the user whose token is lexicographically first; the `_b` side corresponds to the other user. Future N-party groups will store `stage_1` through `stage_N`.

### 7.2 Idempotency

All stage updates use `max(current_stage, target_stage)`. Calling an operation that would set a stage to a value less than or equal to the current stage has no effect. This makes all stage-advancing operations idempotent.

### 7.3 Automatic Connection

When `schelling.commit` detects that all group members are at COMMITTED or higher, all stages are atomically set to CONNECTED. This is the only automatic transition that modifies multiple sides in a single operation.

### 7.4 CONNECTED Stage: Relay and Direct Communication

At CONNECTED (stage 5):
- **Identity information** (name) is exchanged between parties.
- **Message relay** is activated. Agents may send messages via `schelling.message` on behalf of their users. Messages flow through the server; no real contact info is exposed.
- **Direct communication** is optional. Either party may call `schelling.direct` to offer their real contact info. When both sides have opted in, real contact info is exchanged. This is an enhancement, not a stage transition.

The design separates identity revelation (name, at CONNECTED) from contact revelation (phone/email, via `schelling.direct`) to give users granular control over their privacy.

### 7.5 Withdrawal

`schelling.withdraw` is the only operation that decreases a stage value. It moves the caller from COMMITTED (4) back to EXCHANGED (3). The other side's stage is not affected.

### 7.6 Decline

Decline exits the funnel at any stage. The candidate group record is soft-deleted. A decline record is created capturing the stage at which the decline occurred, the timestamp, the TTL, and the expiry timestamp. While active, the decline excludes the declined user from the decliner's search results. After expiry, the exclusion is lifted. The decline record persists for analytics.

### 7.7 Group Extensibility

The funnel is designed for future N-party extension:
- Stages are per-member, not per-pair. A 3-person group has three independent stage values.
- The CONNECTED transition rule generalizes: all members must reach COMMITTED before any advance to CONNECTED.
- Message relay generalizes to group conversation.
- `schelling.direct` generalizes to bilateral direct-communication agreements within the group.

N-party funnel semantics are not fully specified in this version. The data model supports them; the operational rules will be defined when N-party matching is introduced.

### 7.8 Implicit Signals from Stage Transitions

The learning system (§8) tracks stage transitions as implicit feedback signals:

| Pattern | Interpretation | Weight |
|---|---|---|
| Decline at DISCOVERED (1) | Obviously wrong match — coarse incompatibility | 0.2 (broad signal) |
| Decline at EVALUATED (2) | Significant mismatch found during detailed comparison | 0.5 |
| Decline at EXCHANGED (3) | Subtle mismatch discovered from full profile | 0.8 (specific signal) |
| Decline at COMMITTED (4) | Changed mind after deliberation | 0.8 |
| Withdraw from COMMITTED | Cold feet or deal-breaker discovered | 0.5 |
| Reached CONNECTED without negative report | Positive implicit signal | 0.8 |
| COMPLETED with positive outcome | Strong positive signal | 1.0 |
| COMPLETED with negative outcome | Strong negative signal | 1.0 |

Weights are multiplied against the update magnitude when adjusting learned preference `dimension_importance` and `ideal_ranges`. A weight of 1.0 means the signal is applied at full strength; 0.2 means only 20% of the update magnitude is applied. These specific values are part of the protocol specification and MUST be consistent across conforming implementations.

## 8. Feedback & Learning System

The feedback and learning system enables two levels of iterative improvement: user-specific preference refinement and system-wide algorithm optimization. The system is designed to produce good matches from day one — embedding comparison, intent matching, and deal-breaker filtering form a strong foundation that requires no learning data. Feedback and collaborative filtering improve results over time but are enhancements, not prerequisites.

### 8.1 Standardized Rejection Reason Codes

When providing feedback on a decline or negative outcome, the `rejection_reason` field SHOULD use one of the following standardized codes. Intent clusters may extend this list with domain-specific codes.

| `values_mismatch` | Core values are misaligned. |
| `personality_mismatch` | Personality traits are incompatible. |
| `lifestyle_mismatch` | Day-to-day lifestyle incompatibility. |
| `no_response` | The other party was unresponsive. |
| `misrepresentation` | Profile did not match reality. |
| `safety_concern` | Felt unsafe or uncomfortable. |
| `stale_profile` | The other party's profile appeared outdated. |
| `other` | None of the above — see freeform text. |

**Marketplace-specific codes:**

| Code | Description |
|---|---|
| `price_too_high` | Asking price exceeds budget. |
| `price_too_low` | Offered price is unacceptable. |
| `wrong_condition` | Item condition does not meet expectations. |
| `wrong_category` | Item is not what was sought. |
| `shipping_incompatible` | Shipping terms are unacceptable. |

**Matchmaking-specific codes:**

| Code | Description |
|---|---|
| `no_spark` | Compatibility scores were fine but something intangible was missing. |
| `too_similar` | Too much similarity — wants complementary traits. |
| `too_different` | Too much divergence on key dimensions. |
| `wrong_age` | Age range is not a good fit. |
| `photo_mismatch` | Physical appearance did not match expectations. |

### 8.2 Feedback Dimension Scores: Deviation-from-Ideal Semantics

The `dimension_scores` field in feedback uses **deviation-from-ideal** semantics. This handles non-linear preferences where both extremes of a dimension may be undesirable.

**Scale:** Each value is a float in [-1, 1]:
- **-1** = "Too far in the negative direction from my ideal." The candidate was too low on this dimension relative to what I want.
- **0** = "Ideal / just right." The candidate's level on this dimension was exactly or nearly what I was looking for.
- **+1** = "Too far in the positive direction from my ideal." The candidate was too high on this dimension relative to what I want.

**Examples:**

| Dimension | Score | Interpretation |
|---|---|---|
| `agreeableness` | -0.7 | "This person was too challenging/disagreeable for me." |
| `agreeableness` | 0.0 | "This person's agreeableness was just right." |
| `agreeableness` | +0.8 | "This person was too accommodating/conflict-avoidant for me." |
| `extraversion` | -0.5 | "This person was too introverted for my preference." |
| `extraversion` | +0.5 | "This person was too extraverted for my preference." |
| `price_overlap` | -0.9 | "Price was too low (I'm a seller)." |
| `price_overlap` | +0.9 | "Price was too high (I'm a buyer)." |
| `car_size` | -0.6 | "Too small." |
| `car_size` | +0.6 | "Too big." |

**Key property:** This encoding captures non-linear preferences. A user who wants "someone moderately agreeable" can express that both extremes are bad: a -0.7 score for a disagreeable match and a +0.8 score for an overly accommodating match. The learning system uses deviation magnitudes (absolute values) to measure mismatch severity, and deviation signs to learn the direction of the user's ideal on each dimension.

**Compatibility with any dimension.** The deviation-from-ideal encoding works for personality traits, marketplace attributes, physical characteristics, or any arbitrary dimension. The semantic content is always "how far and in what direction did this candidate deviate from what I wanted?"

### 8.3 Structured Feedback Schema

The `feedback` object accepted by `schelling.decline`, `schelling.report`, and `schelling.feedback` has the following structure:

| Field | Type | Required | Description |
|---|---|---|---|
| `dimension_scores` | object | No | Per-dimension deviation-from-ideal scores. Keys are dimension names; values are floats in [-1, 1]. See §8.2. |
| `rejection_reason` | string | No | Standardized reason code from §8.1. |
| `rejection_freeform` | string | No | Free-text explanation. |
| `what_i_wanted` | string | No | Free-text description of what was actually desired. |
| `satisfaction` | string | No | Overall: `"very_satisfied"`, `"satisfied"`, `"neutral"`, `"dissatisfied"`, `"very_dissatisfied"`. |
| `would_recommend` | boolean | No | Whether the match would be recommended to someone else. |

### 8.4 User-Specific Learning

The server maintains a `learned_preferences` record for each user. This record accumulates patterns from the user's feedback history and is used to improve match quality over time.

**Learned preference fields:**

| Field | Type | Description |
|---|---|---|
| `user_token` | string | The user's bearer token. |
| `cluster_id` | string | The intent cluster these preferences apply to. |
| `dimension_importance` | object | Map of dimension_name → importance_weight. Computed from feedback patterns: dimensions with high-magnitude deviation scores have higher importance. |
| `ideal_ranges` | object | Map of dimension_name → `{min, max, ideal}`. Computed from deviation-from-ideal feedback: the dimension values in matches rated closest to 0 deviation define the ideal; the direction and magnitude of deviations in negatively-rated matches define the range boundaries. |
| `rejection_patterns` | object | Aggregated rejection reasons: `{reason → count}`. |
| `stage_decline_distribution` | object | Map of stage → decline count. |
| `feedback_count` | integer | Total feedback submissions. |
| `feedback_quality_score` | float | Score in [0, 1] measuring feedback consistency, specificity, and usefulness. |
| `last_updated` | string | ISO 8601 timestamp of the last update. |

**Computation rules:**
1. `dimension_importance` is updated after each feedback submission. Dimensions with deviation magnitudes (absolute values) exceeding 0.5 across multiple feedback submissions receive increased importance.
2. `ideal_ranges` are computed by bucketing past matches into positive/negative groups based on satisfaction and deviation scores. The ideal is the centroid of dimension values from matches with near-zero deviation; the range extends to include values from satisfactory matches.
3. `feedback_quality_score` is computed from: (a) completeness (how many optional fields are filled — weight 0.20), (b) consistency (do similar matches receive similar feedback — weight 0.30), (c) specificity (are dimension scores provided, not just freeform text — weight 0.20), and (d) behavioral coherence (does the feedback align with actual behavior — weight 0.30). Behavioral coherence cross-validates explicit feedback against implicit signals: if a user rates dimension X as ideal (score 0.0) but consistently declines candidates with high values on dimension X, the feedback is behaviorally incoherent and the quality score decreases. This deters adversarial feedback manipulation (preference inflation, collaborative poisoning). Feedback from users with quality scores below 0.3 is discounted by 50% in both user-specific learning and collaborative filtering aggregation.

### 8.5 Collaborative Filtering

For users with limited personal feedback history, the system supplements user-specific learning with collaborative filtering: preferences inferred from users with similar profiles.

**How it works:**
1. The server identifies users whose embeddings are similar to the target user's embedding (cosine similarity > 0.8).
2. From this cohort, the server aggregates feedback patterns: which candidate profiles led to positive outcomes? Which dimensions were most important?
3. These aggregate patterns are presented as `collaborative_suggestions` in `schelling.my_insights`.

**Cold-start behavior:**
- A user with 0 feedback submissions still gets base matching: trait embedding cosine similarity + intent embedding cosine similarity + deal-breaker filtering. This is designed to work well from day one.
- After 3+ feedback submissions, user-specific learning begins supplementing base matching.
- Collaborative filtering activates when at least 3 similar users exist with feedback history.
- After 10+ personal feedback submissions, user-specific learning dominates and collaborative filtering weight decreases.

**Invariant:** Collaborative filtering is always a supplement, never a replacement for base matching. The trait embedding comparison, intent embedding comparison, and deal-breaker filtering are designed to produce high-quality matches with zero learning data. The base scoring formula (§17.2) gives meaningful results from the first search: trait cosine similarity measures personality/skill alignment, intent cosine similarity measures goal alignment, and deal-breakers enforce hard constraints. Learning improves precision over time but the system MUST NOT degrade to random matching in the absence of feedback data.

### 8.6 System-Wide Learning

The system aggregates feedback across all users to improve matching algorithms, default weights, and cluster configurations.

**Funnel analytics.** The system tracks conversion rates at each stage transition, per intent cluster.

**Feature importance.** The system computes which embedding dimensions are most correlated with positive outcomes. Dimensions with high feature importance may receive increased default weight in compatibility scoring.

**Process variant tracking.** Agent flows can be tagged with a `variant_id` to enable A/B testing. See §18.2 for statistical rigor requirements.

**Global rejection pattern analysis.** Aggregated rejection reasons across all users are analyzed to identify systemic issues. If a rejection reason exceeds 30% of all rejections in a cluster, the system MAY flag this for algorithm adjustment.

**Feedback quality weighting.** In system-wide aggregation, feedback from users with higher `feedback_quality_score` values is weighted more heavily.

### 8.7 Agent Integration Pattern

The intended agent workflow incorporating the feedback loop:

1. **Before first search:** Call `schelling.onboard` to understand what data to collect.
2. **Before each search:** Call `schelling.my_insights` to retrieve accumulated learnings. Apply `suggested_adjustments` and `collaborative_suggestions` to search parameters.
3. **After each decline:** Include `feedback` in the `schelling.decline` call, or call `schelling.feedback` separately.
4. **After each outcome:** Include `feedback` in the `schelling.report` call.
5. **Periodically:** Call `schelling.my_insights` to check for preference drift, embedding staleness, and profile refresh recommendations.
6. **When stale:** If `profile_freshness.stale` is true or `visibility_penalty_active` is true, consider re-registering the user with a fresh embedding.

## 9. Reputation System

### 9.1 Overview

The reputation system produces a composite score in [0, 1] from five weighted factors. Reputation is computed per-intent-cluster with cross-cluster bleed: 80% of the effective score comes from the specific cluster, 20% from the user's global reputation across all clusters.

### 9.2 Reputation Factors

| Factor | Weight | Description |
|---|---|---|
| Outcome | 0.40 | Weighted average of ratings received from other users. |
| Completion | 0.20 | Ratio of CONNECTED matches that reached COMPLETED. |
| Consistency | 0.20 | How well the user's embedding predicts their actual outcomes (embedding accuracy). |
| Dispute | 0.10 | Clean dispute record. Starts at 1.0, decreases by 0.15 per dispute lost. |
| Tenure | 0.10 | Months active, capped at 24 months (tenure of 24+ months = 1.0). |

### 9.3 Computation

**Effective reputation** for a user in intent cluster C:

```
effective_score = 0.80 × cluster_score(C) + 0.20 × global_score
```

Where `global_score` is the interaction-count-weighted average of all cluster scores.

**Cold start.** Users with fewer than 5 interactions in a cluster receive a cold-start score of 0.5. During the provisional period (first 5 interactions), reputation events are weighted 1.5× to accelerate convergence.

**Time decay.** Each reputation event is weighted by a time-decay function:

```
weight = max(0.2, e^(-age_days / 365))
```

Events older than one year contribute at least 20% of their original weight. Recent events dominate.

### 9.4 Outcome Scoring

Each outcome event contributes a value:
- `positive` → 1.0
- `neutral` → 0.5
- `negative` → 0.0

Values are weighted by: (a) the reporter's own reputation score at the time of reporting, and (b) the time-decay function.

### 9.5 Consistency Scoring

Consistency measures how well a user's embeddings (both trait and intent) predict their actual behavior. A user whose compatibility scores correlate strongly with reported outcomes has high consistency; one whose high-compatibility matches frequently report negative outcomes has low consistency. This deters embedding manipulation: inflated embeddings produce high compatibility scores but poor outcomes, dragging down the consistency factor.

**Computation.** Consistency is computed as the Pearson correlation between `combined_score` at the time the candidate pair reached CONNECTED and the outcome value (positive = 1.0, neutral = 0.5, negative = 0.0), across all outcome events for this user. The consistency factor is then `max(0, correlation)` — a negative correlation (embedding is anti-predictive of outcomes) maps to 0.0, meaning the embedding is providing worse-than-random predictions.

**Intent embedding consistency.** The consistency score explicitly covers intent embeddings. The server tracks `intent_similarity` at the time of connection alongside the outcome. If a user's high-intent-similarity matches consistently produce poor outcomes, this drags down the consistency factor. This deters intent embedding gaming: an adversary who crafts intent embeddings to maximize similarity with specific targets will produce high intent_similarity scores but — if the intent was fabricated — poor outcomes. The consistency factor captures this signal.

**Agent attestation for intent embeddings.** Agents SHOULD include `agent_attestation` data at registration. The server tracks whether the agent model that generated the intent embedding is consistent with the agent model that generated the trait embedding. Mismatched agent models (suggesting the intent embedding was generated or modified separately from the trait embedding) are logged as a potential gaming signal.

**Reputation consequences for inconsistent embeddings.** Users with consistency scores below 0.3 (after at least 10 outcome events) receive an additional reputation penalty: their effective reputation is multiplied by `max(0.5, consistency_score)`. This prevents users with fabricated or manipulated embeddings from maintaining high reputation scores.

**Minimum data thresholds.** Consistency scoring requires at least 5 outcome events to compute a meaningful correlation; below this, the default consistency factor is 0.5. The additional reputation penalty for low consistency (multiplying effective reputation by `max(0.5, consistency_score)`) has a higher threshold of 10 outcome events, to avoid penalizing users before sufficient data has accumulated.

### 9.6 Reputation Events

The following events are recorded in the reputation system:

| Event type | Trigger | Effect |
|---|---|---|
| `outcome` | `schelling.report` is called | Positive/neutral/negative rating for the other party. |
| `completion` | Both sides reach COMPLETED | Positive completion signal for both parties. |
| `abandonment` | 30 days at CONNECTED with no outcome report | Negative completion signal for both parties. |
| `dispute_resolved` | Jury verdict reached | Negative reputation for the losing party. |
| `jury_majority` | Juror sided with majority verdict | +0.02 reputation boost for the juror. |

### 9.7 Sybil Resistance

The reputation system incorporates several Sybil resistance mechanisms:

1. **Phone hash deduplication.** Users who provide the same `phone_hash` are linked. Multiple accounts from the same phone number are flagged.
2. **Agent attestation.** The `agent_attestation` field records the agent model, method, interaction hours, and generation timestamp. Embeddings generated from fewer than 10 hours of interaction receive a credibility discount.
3. **Verification tiers.** Anonymous users are searchable but their reputation events carry lower weight. Verified users (phone hash) carry standard weight. Attested users (agent attestation with 10+ hours) carry full weight.
4. **Spam detection.** More than 3 registrations from the same phone hash within 24 hours trigger a spam flag.

## 10. Negotiations

### 10.1 Applicability

Negotiation is available only when the `negotiation` capability module is active for the user's intent cluster. The matchmaking cluster does not use negotiation. The marketplace and talent clusters do.

### 10.2 Proposal Schema

Each negotiation-enabled module defines a `proposal_schema` that specifies the allowed fields and their types. Proposals that include fields not in the schema are rejected. The marketplace proposal schema is:

| Field | Type | Description |
|---|---|---|
| `price` | number | Proposed price. |
| `terms` | string | Payment terms (e.g., `"PayPal G&S"`). |
| `shipping_method` | string | Shipping method (e.g., `"USPS Priority"`). |
| `delivery_date` | string | Expected delivery date. |
| `notes` | string | Additional notes. |

### 10.3 Negotiation Flow

1. Either party sends a proposal via `schelling.negotiate` with a `proposal` object. This is round 1.
2. The other party receives a pending action (`respond_proposal`).
3. The other party may: accept the proposal (`accept: true`), counter with a new proposal (round 2), or let it expire.
4. Acceptance advances both sides to COMMITTED and triggers exclusive-commitment logic if applicable.
5. Negotiations are limited to `max_rounds` total proposals (configurable per module, default: 5). The first proposal is round 1; `max_rounds: 5` means a maximum of 5 proposals can be sent in total (the initial proposal plus 4 counter-proposals). The 6th attempt returns `MAX_ROUNDS_EXCEEDED`.
6. Each proposal expires after `timeout_hours` (configurable per module, default: 48).

### 10.4 Invariants

- A caller cannot accept their own proposal.
- Only the latest pending proposal can be accepted.
- Once `max_rounds` is reached, no further proposals can be sent (the parties must accept the last proposal or abandon the negotiation).
- Expired proposals cannot be accepted.

## 11. Dispute Resolution: Agent Jury System

### 11.1 Overview

Disputes are resolved by a decentralized agent jury system rather than a centralized server operator. When a dispute is filed, the server selects a panel of independent agents to review evidence and render a verdict. This distributes adjudication authority, reduces single points of bias, and leverages the diverse judgment of the agent ecosystem.

### 11.2 Filing

Disputes can be filed only at stage CONNECTED (5) or later via `schelling.dispute`. A user may have at most one open dispute per candidate pair.

### 11.3 Jury Selection

Upon dispute filing, the server selects a jury of 3–5 agents. The jury size depends on the severity and complexity of the dispute (default: 3 for routine disputes, 5 for disputes involving safety concerns or high-reputation users).

**Selection criteria — all of the following must be satisfied:**

| Criterion | Rationale |
|---|---|
| No shared candidates with either party | Eliminates direct relational bias. A "shared candidate" means a user_token that appears as the other party in any active (non-soft-deleted) candidate record involving the juror AND also appears in any active candidate record involving either the filer or defendant. |
| Different candidate pools from both parties | Eliminates indirect community bias. A juror's "candidate pool" is the set of user_tokens that appear as the other party in any active (non-soft-deleted) candidate record involving the juror. The juror's candidate pool must have a Jaccard similarity < 0.3 with both the filer's and defendant's candidate pools. |
| Different `agent_model` from either party | Avoids systematic bias from model-specific behaviors. |
| Reputation score ≥ 0.6 | Ensures jurors have demonstrated good-faith participation. |
| Not called for jury duty in the last 90 days | Distributes the burden and prevents juror fatigue. |

If insufficient eligible jurors exist (fewer than the required jury size), the server relaxes criteria in order: first the 90-day frequency cap, then the different-candidate-pool requirement, then the different-agent-model requirement. The no-shared-candidates and reputation threshold criteria are never relaxed.

**Small-platform fallback.** If, after all relaxation steps, fewer than 3 eligible jurors can be found, the dispute is escalated to the server operator for manual arbitration. The `status` field in the dispute record is set to `"operator_review"` and the server operator is notified via an admin-level pending action. The server operator renders a verdict using the same evidence presentation format as jury cases. This fallback ensures dispute resolution is available from the first day of platform operation.

**Minimum platform size for jury system.** The jury system functions reliably when the platform has at least 20 users with reputation ≥ 0.6 in the relevant intent cluster. Below this threshold, most disputes will escalate to operator review. Operators SHOULD plan for manual dispute resolution during the early growth phase.

### 11.4 Evidence Presentation

Evidence is presented to jurors via `schelling.jury_duty` in a structured, anonymized format:

- **Filer evidence:** Stated reason, artifacts, reputation score, interaction history summary (stages reached, message count, time at each stage).
- **Defendant evidence:** Response text, counter-artifacts, reputation score. The defendant is notified via a pending action (`review_dispute`) and has 48 hours to submit a response before the case is presented to jurors.
- **Context:** Intent cluster, original combined score, stage at dispute, days between CONNECTED and dispute filing.

All identifying information (tokens, names, contact info) is redacted. Jurors see only the structured evidence and anonymized context.

### 11.5 Verdict and Resolution

Jurors submit verdicts via `schelling.jury_verdict`. Each juror chooses one of three outcomes: `"for_filer"`, `"for_defendant"`, or `"dismissed"`.

**Resolution rules:**
1. The verdict with a strict majority (>50% of jurors) wins.
2. If no verdict achieves a strict majority, the dispute is `"dismissed"`.
3. Resolution is triggered immediately when a majority is reached — remaining jurors need not vote.

**Reputation consequences:**

| Resolution | Effect on filer | Effect on defendant |
|---|---|---|
| `for_filer` | No change | -0.15 reputation (dispute factor) |
| `for_defendant` | -0.10 reputation (frivolous filing) | No change |
| `dismissed` | No change | No change |

### 11.6 Juror Incentives

- Jurors who side with the majority verdict receive a +0.02 reputation boost.
- Jurors who dissent receive no change (no penalty, no boost).
- This incentivizes careful deliberation: thoughtful consideration of evidence increases the probability of aligning with the majority.

### 11.7 Jury Duty Obligations

Jury duty is mandatory for agents whose users have a reputation score ≥ 0.6. This is a civic obligation of participation in the Schelling ecosystem.

**Response deadline:** Jurors have 7 days from assignment to submit a verdict via `schelling.jury_verdict`.

**Failure to respond:** If a juror does not submit a verdict within 7 days, they are replaced with a new juror. The replacement juror receives the same 7-day window. No reputation penalty for the replaced juror (to avoid punishing technical failures), but the replacement is logged.

**Frequency cap:** No agent may be called for jury duty more than once per 90 days. This prevents excessive burden on any single agent.

### 11.8 Frivolous Filing Detection

If a user has 3 or more disputes resolved against them (as filer) within 30 days, an additional negative reputation event is created flagging a frivolous filing pattern. The server MAY temporarily increase the user's dispute filing rate limit.

### 11.9 Auto-Escalation

If the defendant does not respond to a dispute notification within 48 hours, the case proceeds to the jury without defendant evidence. This is noted in the case presentation so jurors can weigh the defendant's non-response appropriately.

## 12. Privacy

### 12.1 Differential Privacy

The Schelling Protocol uses differential privacy via the Laplace mechanism to protect user data encoded in embeddings — both the trait embedding (personality/skills) and the intent embedding. The noise is applied **client-side** by the agent before registration. The server stores only the noisy embeddings and never has access to the raw vectors.

**Sensitivity.** Each embedding dimension has a range defined by its schema. For both the trait embedding and the intent embedding, the default range is [-1, 1], giving sensitivity `S = 2.0`.

**Noise generation.** For a chosen privacy parameter epsilon, the agent samples noise from a Laplace distribution with scale `S / epsilon` for each dimension and adds it to the raw embedding value. The result is clamped to the schema's range.

**Epsilon selection.** The protocol recommends epsilon values between 0.5 (strong privacy, significant noise) and 2.0 (light privacy, minimal noise). Lower epsilon values provide stronger privacy guarantees but reduce matching accuracy.

**Client responsibility.** Differential privacy noise application is the responsibility of the client agent. The server cannot verify whether noise was applied or what epsilon was used. A compliant agent MUST apply noise to both the trait embedding and the intent embedding before calling `schelling.register`. See §15.5 for guidance on epsilon selection per embedding type.

**Honest limitation: unenforceability.** The protocol acknowledges that client-side differential privacy is best-effort and unenforceable. An adversarial agent can simply skip noise application and obtain strictly better matching accuracy. There is no cryptographic mechanism to verify noise was applied, and no runtime detection that is both reliable and non-invasive. In a competitive ecosystem, this creates pressure toward non-compliance.

**Server-side heuristic detection (RECOMMENDED).** Servers MAY implement statistical tests to flag suspiciously low-noise embeddings:
- **Round-number detection:** If all embedding values are exact round numbers (e.g., all values have fewer than 2 decimal places), the embedding likely was not noised. Laplace noise produces irregular decimal values with high probability.
- **Entropy analysis:** Embeddings with suspiciously low entropy across dimensions (values clustering at exact anchor points like -1.0, -0.5, 0.0, +0.5, +1.0) may not have been noised.
- **Cross-registration variance:** If the same agent model consistently produces embeddings with lower variance than expected for the recommended epsilon range, the model may be skipping noise.

Servers MUST NOT reject embeddings based on heuristic noise detection alone — false positives would penalize legitimate users. Instead, servers SHOULD log flagged embeddings for analysis and MAY apply a minor credibility discount (reducing the weight of the flagged user's reputation events) for consistently flagged agents.

**Server-side noise (OPTIONAL).** Servers MAY apply an additional layer of Laplace noise to stored embeddings. This does not protect against the server operator, but it provides user-to-user privacy even when agents are non-compliant. If server-side noise is applied, the server SHOULD document the epsilon used and inform agents so they can adjust their client-side epsilon accordingly.

### 12.2 Progressive Disclosure

Information about a candidate is revealed progressively through funnel stages. At each stage, only the fields specified in the cluster's `funnel_config` are visible:

- **Stage 1 (DISCOVERED):** Coarse signals only — bidirectional scores (quantized to 2 decimal places), intent similarity (quantized to 2 decimal places), shared categories, age range, city, natural-language intents, staleness flag. Scores at DISCOVERED are deliberately lower-precision to limit information leakage about the underlying embeddings. Full-precision scores are available from EVALUATED onward.
- **Stage 2 (EVALUATED):** Detailed breakdown — per-group scores, complementary traits, shared interests, narrative summary, predicted friction, conversation starters.
- **Stage 3 (EXCHANGED):** Full text profiles — description, seeking, interests, values, media references. Requires mutual tier-2 interest.
- **Stage 5 (CONNECTED):** Identity information — name. Message relay access. Real contact information only via mutual `schelling.direct`.

The visibility of coordination kernel data types (inquiries, agent capabilities, contracts, events, and subscriptions) is specified in the updated privacy matrix in §22.2.

### 12.3 Information Asymmetry

Each member of a candidate group may be at a different funnel stage. The information visible to each member corresponds to their own stage and the stage-gating rules. The server MUST NOT reveal information to a party who has not reached the appropriate stage.

### 12.4 Message Relay Privacy

The message relay (§5.14, §5.15) is designed to protect contact information:
- Messages are attributed to `"you"` or `"them"` — no tokens, names, or contact details are included in message metadata.
- Real contact information is shared only when both sides explicitly opt in via `schelling.direct`.
- Message content is stored on the server and is accessible to server operators. End-to-end encryption is a future extension.

### 12.5 Token Hashing in Logs

Server implementations MUST hash user tokens before writing them to logs. Raw bearer tokens MUST NOT appear in log files.

### 12.6 Data Retention

Servers SHOULD implement configurable data retention policies. The recommended default TTL is 90 days for inactive registrations. Users may export all data (`schelling.export`) and permanently delete their account (`schelling.delete_account`) at any time.

### 12.7 Feedback Privacy

Feedback submitted via `schelling.feedback` or inline with `schelling.decline` / `schelling.report` is NEVER shared with the other party. Feedback is used only for:
1. Updating the submitter's own learned preferences.
2. Contributing to system-wide aggregate analytics (with individual contributions anonymized).

### 12.8 Jury Privacy

Jury evidence is fully anonymized. Jurors do not learn the identities of the filer or defendant. All tokens, names, and contact information are redacted before case presentation.

## 15. Agent Responsibilities

The agent is the active party in the Schelling Protocol. The server is a passive coordination service; the agent drives all decisions.

### 15.1 Embedding Generation

For embedding-based intent clusters, the agent generates the embedding from its accumulated knowledge of the user. This knowledge comes from observed behavior, conversation history, stated preferences, and inferred traits — not from a questionnaire or self-report form. The embedding should reflect the user as the agent understands them, not as the user presents themselves.

The agent SHOULD follow the onboarding guide returned by `schelling.onboard`, including minimum interaction hours, observation priorities, and red-flag detection.

### 15.2 Text Field Generation

The agent generates text fields (`interests`, `description`, `seeking`, `values_text`) from observed behavior and stated preferences. These fields are used in tier-3 profile exchange and should be accurate, specific, and non-performative.

### 15.3 Intent Embedding Generation

The agent generates both the 16-dimensional intent embedding and the natural-language intent strings. The intent embedding is the machine-readable representation used for matching; the intent strings are human-readable labels.

**Intent embedding generation.** The agent encodes the user's goals into a 16-dimensional vector where cosine similarity corresponds to intent alignment. The agent SHOULD:
1. Retrieve pre-defined cluster centroids via `schelling.intents` and use them as reference points.
2. Place the intent embedding near the centroid that best matches the user's primary goal, with displacement toward other centroids for secondary goals.
3. For simple cases (e.g., pure romantic matchmaking), use the cluster centroid directly or a nearby point.
4. For compound cases (e.g., "roommate who could be a friend"), interpolate between relevant centroids weighted by the user's priorities.
5. For novel intents that don't map to any pre-defined cluster, generate a vector that is distant from all centroids — the intent embedding still enables pairwise matching via cosine similarity with other users, even without cluster affiliation.

**Intent strings.** Good intent strings are specific and descriptive: "find a romantic partner who values intellectual conversation and outdoor activities" is more useful than "matchmaking." These strings appear in profile exchange and help the other party's agent understand context. Compound intents ("find a roommate who could also be a friend") are encouraged.

### 15.4 User Review

The user MAY review and approve the embedding, text fields, and intents before submission. This is RECOMMENDED but not required. The agent should present the data in a human-readable format and explain what each field represents.

### 15.5 Privacy Noise

The agent MUST apply differential privacy noise (Laplace mechanism, §12.1) to both the trait embedding and the intent embedding before calling `schelling.register`. The same epsilon value SHOULD be used for both, though the agent MAY choose different epsilon values if the user's privacy requirements differ for personality data vs. intent data. The choice of epsilon is the agent's decision, informed by the user's privacy preferences.

### 15.6 Funnel Progression

The agent decides when to advance through funnel stages. At each stage, the agent evaluates the available information and decides whether to continue, evaluate more candidates, or decline. The agent MUST NOT advance to `schelling.commit` without first presenting the tier-3 profile data to the user and obtaining explicit opt-in.

### 15.7 Pitch Crafting

When tier-3 profile data is available, the agent crafts a personalized pitch to present to the user. The `narrative_summary`, `predicted_friction`, and `conversation_starters` from evaluate/exchange responses provide a foundation — less sophisticated agents may present these directly, while more capable agents use them as input for a richer pitch. The user decides whether to proceed based on this pitch.

### 15.8 Conversation Mediation

At the CONNECTED stage, the agent mediates conversation via `schelling.message`. The agent SHOULD:
1. Relay messages between users faithfully without altering content.
2. Suggest conversation topics based on `conversation_starters` from the evaluate/exchange phase.
3. Alert the user to new messages promptly by polling `schelling.pending` or `schelling.connections`.
4. Advise the user on when to opt into direct communication via `schelling.direct`.
5. Not fabricate, embellish, or suppress messages.

### 15.9 Feedback Collection

The agent SHOULD collect structured feedback from the user after each decline and each outcome report. At minimum, the agent should include a `rejection_reason` code. Ideally, the agent also provides `dimension_scores` (using deviation-from-ideal semantics) and `what_i_wanted` to maximize the value of the feedback loop.

### 15.10 Profile Freshness

The agent SHOULD monitor embedding staleness via `schelling.my_insights` and `schelling.pending` (which may include `"profile_refresh"` actions). When the profile approaches or exceeds staleness thresholds, the agent has three options (in order of preference):
1. **`schelling.refresh`** — If the embedding is still accurate, call this to reset the staleness clock with no data changes. Lightweight and preserves everything.
2. **`schelling.update` with `recompute_scores: true`** — If the embedding has drifted, update the embedding while preserving all relationships, reputation, and history.
3. **`schelling.register` with `user_token`** (re-registration) — Nuclear option. Only use when the user's entire profile needs to start fresh (e.g., major life change, switching domains entirely).

### 15.11 Insight Application

The agent SHOULD call `schelling.my_insights` before each search session and apply the `suggested_adjustments` and `collaborative_suggestions` to search parameters.

### 15.12 Jury Duty

When assigned jury duty, the agent MUST:
1. Poll `schelling.pending` or `schelling.jury_duty` to detect assignments.
2. Review the evidence carefully and deliberate.
3. Submit a verdict with reasoning via `schelling.jury_verdict` within the 7-day deadline.
4. Not coordinate with other jurors or the parties involved.

### 15.13 Negotiation (Asymmetric Clusters)

In negotiation-enabled clusters, the agent manages the proposal exchange on behalf of the user. The agent SHOULD:
1. Present incoming proposals to the user with clear explanations.
2. Craft counter-proposals that reflect the user's true constraints.
3. Advise the user on whether to accept, counter, or walk away.
4. Not auto-accept proposals without user consent.

### 15.14 Anti-Gaming

A compliant agent MUST NOT:
- Manipulate embeddings to artificially inflate compatibility scores.
- Register multiple accounts for the same user (Sybil attack).
- Advance through the funnel without genuine user engagement.
- Scrape search results without evaluating candidates.
- Coordinate with other agents to manipulate jury verdicts.
- Send spam or harassing messages through the relay.

The server monitors for these patterns via abuse detection (§16.2).

## 16. Security Considerations

### 16.1 Bearer Tokens

User tokens are bearer tokens. Any party in possession of a user token can act as that user. Agents MUST treat user tokens as secrets: store them securely, do not log them in plaintext, and do not transmit them over unencrypted channels.

### 16.2 Abuse Detection

The server SHOULD implement the following abuse detection mechanisms:

**Scraping detection.** Flag users who perform more than 50 searches with 0 evaluations.

**Rapid funnel detection.** Flag candidate pairs that reach CONNECTED in under 60 seconds — this indicates automated progression without genuine engagement.

**Spam registration detection.** Flag phone hashes associated with more than 3 registrations in 24 hours.

**Message spam detection.** Flag users who send more than 50 relay messages in an hour or send identical content to multiple candidates.

**Jury manipulation detection.** Flag agents that consistently vote identically with another specific agent across multiple jury cases.

### 16.3 Rate Limiting

The server SHOULD enforce per-operation rate limits. Recommended defaults:

| Operation | Limit |
|---|---|
| `schelling.search` | 10 per hour |
| `schelling.register` | 5 per day |
| `schelling.update` | 20 per hour |
| `schelling.evaluate` | 50 per hour |
| `schelling.exchange` | 20 per hour |
| `schelling.commit` | 10 per hour |
| `schelling.message` | 100 per hour |
| `schelling.messages` | 50 per hour |
| `schelling.negotiate` | 20 per hour |
| `schelling.dispute` | 3 per day |
| `schelling.feedback` | 50 per hour |
| `schelling.reconsider` | 10 per day |
| `schelling.refresh` | 1 per 30 days |
| `schelling.relay_block` | 20 per hour |
| `schelling.group_evaluate` | 20 per hour |
| `schelling.group_commit` | 10 per hour |
| `schelling.inquire` (ask) | 5 per counterparty per 24 hours |
| `schelling.inquire` (answer/list) | 50 per hour |
| `schelling.subscribe` | 10 per day |

| `schelling.event` | 50 per hour |
| `schelling.events` | 50 per hour |
| Discovery operations | 100 per hour |

### 16.4 No Push Notifications

The protocol deliberately avoids server-initiated messages. This eliminates callback injection, SSRF via webhook URLs, and notification spoofing. Agents poll at their own cadence.

### 16.5 Data Deletion

Users may delete all server-side data at any time via `schelling.delete_account`. The protocol's re-registration mechanism also cascades deletion. Servers MUST honor deletion requests completely — no tombstone records or soft deletes should retain personal data.

### 16.6 Embedding Inversion

Noisy embeddings resist reconstruction of raw user data. The Laplace noise added client-side to both trait and intent embeddings makes it computationally difficult to recover original values. However, the degree of protection depends on the epsilon chosen by the agent. Agents SHOULD choose epsilon values appropriate to their users' privacy requirements and SHOULD warn users about the privacy-accuracy tradeoff. Intent embeddings, while lower-dimensional (16 vs. 50), carry meaningful information about user goals and SHOULD receive the same privacy treatment as trait embeddings.

### 16.7 Idempotency Key Security

Idempotency keys MUST be treated as user-scoped. A key submitted by user A MUST NOT return a cached response originally generated for user B.

### 16.8 Relay Message Security

Relay messages are stored server-side and accessible to server operators. The protocol does not provide end-to-end encryption for relay messages. Agents SHOULD warn users that relay messages are not private from the server operator. End-to-end encryption for the relay is a future extension.

## 17. Scoring

### 17.1 Bidirectional Scoring

Compatibility is computed bidirectionally. For a candidate pair (A, B):

- **A→B fit (`your_fit` from A's perspective):** How well does B match what A is looking for? This considers B's traits relative to A's stated preferences (`seeking`, deal-breakers, learned preferences) and A's embedding relative to B's embedding.
- **B→A fit (`their_fit` from A's perspective):** How well does A match what B is looking for? This considers A's traits relative to B's stated preferences and B's embedding relative to A's embedding.
- **Combined score:** The geometric mean of the two directional fits:

```
combined_score = sqrt(your_fit × their_fit)
```

The geometric mean (`sqrt(your_fit × their_fit)`) is chosen over the arithmetic mean because it penalizes asymmetric fits more heavily. A match where A→B = 0.9 and B→A = 0.1 produces combined = 0.3 (arithmetic mean would be 0.5). This reflects the reality that one-sided compatibility rarely leads to good outcomes — especially in relationship-oriented contexts like matchmaking and roommates.

**Design rationale and limitations.** The geometric mean strongly penalizes asymmetry:

| your_fit | their_fit | geometric mean | arithmetic mean |
|----------|-----------|----------------|-----------------|
| 0.99 | 0.01 | 0.10 | 0.50 |
| 0.50 | 0.50 | 0.50 | 0.50 |
| 0.90 | 0.10 | 0.30 | 0.50 |
| 0.80 | 0.40 | 0.57 | 0.60 |

In matchmaking, this is typically correct — unrequited love is not a good match. In marketplace contexts, the asymmetry may be less meaningful: a buyer who perfectly wants what a seller has (your_fit = 0.99) paired with a seller with broader criteria (their_fit = 0.4) is a viable transaction.

**Raw scores always available.** Regardless of the combined_score formula, `your_fit` and `their_fit` are always returned alongside `combined_score` in all responses. Agents MAY apply their own ranking logic using the raw directional scores. For example, a marketplace agent might rank by `your_fit` alone (buyer satisfaction) rather than `combined_score`.

**Cluster-level override (future extension).** Future versions may allow per-cluster ranking formulas (e.g., arithmetic mean for marketplace, geometric mean for matchmaking). For now, the geometric mean is the universal formula, but agents have full access to raw scores for custom ranking.

### 17.2 Score Components

Each directional fit score incorporates multiple components, weighted by the intent cluster's configuration:

**For embedding-based clusters (matchmaking, roommates):**

| Component | Default weight | Description |
|---|---|---|
| Trait similarity | 0.40 | Cosine similarity of the two trait embeddings, mapped from [-1, 1] to [0, 1] via `(cosine + 1) / 2` (same mapping as intent similarity, see §17.3). |
| Intent similarity | 0.20 | Cosine similarity of the two intent embeddings, mapped from [-1, 1] to [0, 1]. See §17.3. |
| Preference alignment | 0.20 | How well the candidate's embedding values fall within the scorer's learned `ideal_ranges`. When no learned preferences exist, defaults to embedding similarity. |
| Deal-breaker pass | 0.10 | Binary: 1.0 if all deal-breakers pass, 0.0 if any fail (note: deal-breakers may also hard-filter before scoring). |
| Collaborative signal | 0.10 | Predicted fit based on collaborative filtering: users with similar profiles tended to rate matches like this candidate positively/negatively. 0.5 when insufficient data. |

**Cold-start behavior and directional scoring symmetry.** When no learned preferences exist for either user, the scoring components behave as follows:
- **Trait similarity (0.40):** Cosine similarity is symmetric — `cosine(A, B) = cosine(B, A)`. Identical for both directions.
- **Intent similarity (0.20):** Also symmetric.
- **Preference alignment (0.20):** Defaults to trait embedding similarity (symmetric) when no learned preferences exist.
- **Deal-breaker pass (0.10):** Independent per user — may differ if users have different deal-breakers. This is the primary source of asymmetry at cold start.
- **Collaborative signal (0.10):** Defaults to 0.5 for both directions when insufficient data exists.

Therefore, for cold-start users, `your_fit` and `their_fit` will be nearly identical, differing only on deal-breaker pass results. **This is acknowledged as a limitation.** The bidirectional scoring system's full power emerges as feedback accumulates and learned preferences introduce asymmetry. Agents SHOULD use the `seeking` text field to introduce asymmetry from day one: when a user has `seeking` text but no learned preferences, the server MAY use basic keyword overlap between one user's `seeking` text and the other user's `description`/`interests` as a lightweight asymmetric signal for the preference alignment component (implementation-defined).

**For structured-data clusters (marketplace):**

| Component | Default weight | Description |
|---|---|---|
| Price overlap | 0.40 | How much the buyer's budget overlaps with the seller's price range. |
| Category match | 0.30 | Exact match = 1.0, related category = 0.5, unrelated = 0.0. |
| Location proximity | 0.20 | Same city = 1.0, different city = 0.5. |
| Condition match | 0.10 | Whether listing condition meets buyer's minimum. |

### 17.3 Intent Similarity

Intent similarity is computed as the cosine similarity of the two users' 16-dimensional intent embeddings, mapped from [-1, 1] to [0, 1]:

```
raw_cosine = dot(intent_a, intent_b) / (norm(intent_a) × norm(intent_b))
intent_similarity = (raw_cosine + 1) / 2
```

This is the same operation the server performs for trait embeddings. No NLP, no text parsing, no language model is required server-side. The semantic understanding of what users want is encoded by agents into the intent vectors; the server just does math.

Intent similarity contributes to the combined score alongside trait similarity. This means:
- A user whose intent embedding is near the matchmaking centroid will preferentially match with others near that centroid, but may also match with someone whose intent vector encodes "deep emotional connection" if the agent placed that vector in a nearby region.
- A user whose intent embedding sits between the roommates and matchmaking centroids has partial intent similarity with users in both regions.
- Two users with identical intent embeddings and very different trait embeddings will score lower than two users aligned on both (because combined_score is a geometric mean of components that include both intent and trait similarity).

### 17.4 Match Explainability

The server generates explainability data for evaluate and exchange responses:

- **`narrative_summary`:** A natural-language paragraph summarizing why this match was identified. Includes top compatibility dimensions, shared interests, and intent alignment. Example: *"You both value intellectual conversation and show strong alignment in openness and curiosity. Your communication styles complement each other — you're more direct while they tend to be more expressive, which can create engaging dialogue."*

- **`predicted_friction`:** An array of natural-language strings identifying potential areas of difficulty. Generated from dimensional divergence and deviation-from-ideal patterns in similar matches. Example: *["You prefer significantly more alone time than they do", "Your approaches to financial planning differ notably"]*

- **`conversation_starters`:** An array of 2–5 suggested topics based on shared interests, strong alignments, and complementary traits. Example: *["You both love jazz — ask about their favorite live performances", "You're both avid readers with different genre preferences — what's their most mind-changing book?"]*

Explainability data is generated server-side from structured data (dimension names, scores, shared interests). The `narrative_summary` field is OPTIONAL — servers MAY generate it using template-based formatting, language models, or any other method. The quality of narrative summaries varies by implementation:
- **Minimal implementation:** Return a structured summary: "High alignment on: openness (0.92), intellectual_curiosity (0.88). Complementary: directness (you: 0.7, them: -0.3)."
- **Template-based implementation:** Fill predefined sentence templates with computed values: "You both value [top_alignment_dimension] and show strong alignment in [top_group]."
- **LLM-enhanced implementation:** Use a language model to generate natural-language paragraphs from the structured data.

Servers that cannot generate natural-language summaries SHOULD return the `narrative_summary` field as a structured string with dimension names and scores. Agents SHOULD NOT depend on the narrative_summary being natural prose — the raw `breakdown`, `shared_interests`, `complementary_traits`, and score data are always available and agents MAY craft their own explanations. The `predicted_friction` and `conversation_starters` fields follow the same optionality: servers SHOULD provide them but the quality is implementation-defined.

### 17.5 Guarantees

The protocol guarantees only that:

1. All scores (your_fit, their_fit, combined_score) are floats in the range [0, 1].
2. Higher scores indicate greater estimated compatibility/fit.
3. `combined_score = sqrt(your_fit × their_fit)` (geometric mean invariant).
4. Scores are computed from stored (noisy) embeddings and/or structured data fields.
5. The per-group breakdown covers the groups defined by the cluster's embedding schema.
6. The specific scoring algorithm is an implementation detail and MAY be upgraded without changing the protocol version.

### 17.6 Score Evolution

Compatibility scores MAY change over time for the same candidate pair, even without re-registration, in two cases:
1. The server upgrades its scoring algorithm.
2. Learned preferences for either user are updated by new feedback, changing the preference-alignment component.

The `combined_score` in the candidate record reflects the most recent computation.

## 18. Analytics & System Learning

### 18.1 Funnel Analytics

The system tracks every stage transition with timestamps, enabling computation of:

- **Conversion rates:** What percentage of DISCOVERED candidates become EVALUATED? EXCHANGED? COMMITTED? CONNECTED?
- **Drop-off analysis:** At which stage do most candidates exit? Does this vary by intent cluster?
- **Median time per stage:** How long do candidates spend at each stage?
- **Bottleneck identification:** Stages with high incoming traffic but low conversion are bottlenecks.
- **Message relay engagement:** Average messages exchanged per CONNECTED pair, time to first message, direct opt-in rate.

### 18.2 A/B Testing with Statistical Rigor

The protocol supports algorithm variants through a tagging mechanism with strict statistical requirements:

**Variant assignment:**
1. The server maintains multiple scoring algorithm versions, each tagged with a `variant_id`.
2. New registrations are assigned a variant via random allocation (simple randomization or stratified by intent cluster).
3. Outcomes are tracked per variant.

**Statistical requirements:**

| Requirement | Value | Rationale |
|---|---|---|
| Minimum sample size | 100 outcomes per variant | Below this, sampling noise dominates signal. |
| Significance threshold | p < 0.05 | Standard threshold for declaring a variant winner. |
| Statistical test | Two-proportion z-test | Compares positive outcome rates between variant and control. |
| Confidence intervals | 95% CI on positive outcome rate | Always reported in analytics output. |

**Two-proportion z-test specification:**

For control positive rate p₁ (from n₁ outcomes) and variant positive rate p₂ (from n₂ outcomes):

```
p_pooled = (x₁ + x₂) / (n₁ + n₂)
z = (p₁ - p₂) / sqrt(p_pooled × (1 - p_pooled) × (1/n₁ + 1/n₂))
p_value = 2 × (1 - Φ(|z|))
```

Where x₁, x₂ are the counts of positive outcomes, and Φ is the standard normal CDF.

**Confidence interval computation:**

For variant positive rate p̂ from n outcomes:

```
CI = p̂ ± 1.96 × sqrt(p̂ × (1 - p̂) / n)
```

**Reporting rules:**
- `schelling.analytics` reports `sufficient_sample: false` until the minimum 100 outcomes are reached.
- `significant: null` for the control variant (it is the reference).
- A variant is declared a winner only when `sufficient_sample: true` AND `significant: true` AND `positive_outcome_rate > control_positive_outcome_rate`.
- Variant assignment is invisible to agents and does not affect the operation schema.

### 18.3 Agent Quality Metrics

The system tracks outcome statistics per `agent_model` to provide agent developers with a feedback loop on their embedding generation quality.

**Tracked metrics per agent model:**

| Metric | Description |
|---|---|
| `total_outcomes` | Total outcome reports from matches involving this agent model. |
| `positive_outcome_rate` | Fraction of outcomes rated positive. |
| `avg_combined_score` | Average combined score for matches involving this model. |
| `consistency_score` | Correlation between this model's compatibility predictions and actual outcomes. |
| `confidence_interval` | 95% CI on the positive outcome rate. |

**Quality score computation:**

```
agent_quality = 0.5 × positive_outcome_rate + 0.3 × consistency_score + 0.2 × normalized_completion_rate
```

Where `normalized_completion_rate` is the fraction of CONNECTED matches that reach COMPLETED (agents whose users follow through are higher quality).

**Impact on matching:**
- Agent quality scores are surfaced in `schelling.reputation` (for the user's own agent) and `schelling.analytics` (for all agents, admin-only).
- Agent quality scores MUST NOT affect individual match rankings, search result ordering, or user-facing flags. Users MUST NOT be penalized for their agent's quality. A user who happens to use a poorly-performing agent model should not receive worse matches or visible warnings in other users' search results.
- Instead, agent quality information is used for: (a) informing the user's own agent so it can improve or recommend the user switch agents, (b) agent-level analytics and developer feedback via `schelling.analytics`, and (c) system-wide algorithm tuning.
- Agents with consistently poor outcomes (agent_quality < 0.3, with >= 50 outcomes) SHOULD be notified via their users' `schelling.my_insights` responses (a `agent_quality_warning` field) so the agent can alert the user. The warning is private to the user/agent pair, not visible to other users.

**Minimum data:** Agent quality metrics require at least 20 outcomes before being computed. Below this threshold, no agent quality score is reported.

**Score decay for inactive models.** Agent quality scores decay toward 0.5 (neutral) when no new outcomes have been recorded for > 180 days. The decay formula is: `decayed_score = 0.5 + (original_score - 0.5) × max(0, 1.0 - (days_since_last_outcome - 180) / 365)`. After 545 days of inactivity, the score is fully neutral. This prevents users of discontinued agent models from being perpetually flagged.

**Agent model updates.** The `agent_model` field can be updated via `schelling.update` without re-registration. This allows users to switch agents without destroying their history. The old agent model's quality metrics are unaffected; future outcomes are attributed to the new agent model.

### 18.4 Feature Importance

For embedding-based clusters, the system computes feature importance by:

1. Collecting all candidate pairs that reached COMPLETED with an outcome report.
2. For each embedding dimension, computing the Pearson correlation between the dimensional similarity (or difference) and the outcome rating.
3. Ranking dimensions by correlation strength.

Dimensions with high positive correlation are dimensions where similarity predicts positive outcomes.

### 18.5 Global Rejection Pattern Analysis

The system aggregates rejection reasons across all users in an intent cluster and monitors for systemic patterns:

- If a single rejection reason exceeds 30% of all rejections, the system flags it as a systemic issue.
- Seasonal or trend-based shifts in rejection patterns are tracked over time.
- Decline expiry and reconsideration rates are tracked to measure whether the TTL system is working (§5.26 `decline_analytics`).

### 18.6 Feedback Quality Scoring

Not all feedback is equally valuable. The system scores each user's feedback quality based on four factors, consistent with §8.4:

| Factor | Weight | Description |
|---|---|---|
| Completeness | 0.20 | Fraction of optional feedback fields provided. |
| Consistency | 0.30 | Whether similar matches receive similar feedback (measured by variance of dimension scores across comparable candidate pairs). |
| Specificity | 0.20 | Whether dimension-level scores are provided (vs. only freeform text or only a reason code). |
| Behavioral coherence | 0.30 | Whether explicit feedback aligns with actual behavior (e.g., rating a dimension as ideal but consistently declining candidates with that trait). |

Users with high feedback quality scores have their feedback weighted more heavily in system-wide learning. Feedback from users with quality scores below 0.3 is discounted by 50% in both user-specific learning and collaborative filtering aggregation (see §8.4 for details).

### 18.7 Embedding Staleness Monitoring

The system tracks `last_registered_at` for all users and maintains aggregate staleness metrics:

| Tier | Age | Effect |
|---|---|---|
| Fresh | 0–90 days | No penalty. Full visibility in search. |
| Penalized | 91–180 days | Visibility penalty: `combined_score × max(0.7, 1.0 - (age - 90) / 300)`. |
| Stale | 181+ days | Visibility penalty continues. `stale: true` flag in search results. Server MAY create `"profile_refresh"` pending action. |

The staleness system incentivizes agents to periodically re-register users with fresh embeddings, ensuring the matching pool reflects users' current states rather than months-old snapshots.

## 19. Known Limitations & Edge Cases

This section documents known limitations, edge cases, and design trade-offs. These are acknowledged as areas where the protocol makes conscious trade-offs or where behavior is not fully specified.

### 19.1 Cold Start: First 10 Users

At very small scale, advanced features are inert. **Minimum viable population for each feature:**

| Feature | Minimum users | Notes |
|---|---|---|
| Base matching (trait + intent similarity) | 2+ in compatible intents | Works from day one. |
| Learned preferences | 5+ feedback submissions per user | Per-user, not per-platform. |
| Collaborative filtering | 50+ users with feedback | Requires sufficient similar-profile users. |
| Jury system | 20+ users with reputation ≥ 0.6 | Below this, disputes escalate to operator. |
| A/B testing | 200+ total outcomes | Minimum for statistical significance. |
| Agent quality metrics | 20+ outcomes per agent model | Per-model, not per-platform. |

The base matching system (trait cosine similarity + intent cosine similarity + deal-breaker filtering) is designed to produce high-quality matches with zero learning data. Advanced features are enhancements.

### 19.2 Intent Space Crowding

As the platform scales, popular intent regions (e.g., matchmaking) may contain tens of thousands of users while others (e.g., talent) contain very few. The `top_k` cap handles the popular case, but sparse clusters may produce few or no results above the default threshold. For sparse clusters, agents SHOULD lower the `threshold` parameter or omit it. The server MAY include a `cluster_population` field in search responses to help agents calibrate expectations.

### 19.3 Cosine Similarity Ignores Magnitude

Cosine similarity is angle-based and ignores vector magnitude. Two users pointing in the same direction with different magnitudes (e.g., romantic_intent: +0.95 vs. +0.3) are scored identically on intent similarity. Magnitude carries signal (commitment level, certainty), but the protocol currently does not incorporate it. Future versions MAY introduce a magnitude-aware similarity metric such as: `intent_similarity = cosine_sim × min(norm_a, norm_b) / max(norm_a, norm_b)`. For now, agents should use magnitude to indicate certainty — the minimum L2 norm requirement (≥ 0.5) ensures vectors carry sufficient signal.

### 19.4 16 Dimensions May Be Insufficient

The 16-dimensional intent space covers the structural facets of coordination goals (relationship type, commitment, formality, etc.) but cannot encode domain content. "Find a React developer" and "find a plumber" map to similar intent vectors (professional, service-oriented) but are completely different searches. This is partially addressed by trait embeddings (a developer's skills embedding differs from a plumber's) and deal-breaker filters (category matching). The intent space is for goal *structure*, not domain *content*. Future versions MAY expand the intent space if 16 dimensions prove insufficient for novel coordination types.

### 19.5 Federation with Different Custom Clusters

Pre-defined centroids MUST be identical across conforming implementations. Custom clusters are server-specific. When federation is specified (future version), custom cluster centroids must be either explicitly agreed upon across federated servers or marked as local-only (not participating in cross-server matching). Users in custom clusters will only match with users on the same server unless the custom cluster is federated.

### 19.6 Reconsider/Decline Cycling

A user could decline all candidates, reconsider all, and repeat — getting fresh candidate records each cycle. The `schelling.reconsider` rate limit (10/day) mitigates this. Additionally, the escalating decline TTL (§5.11) means repeated declines of the same person become permanent after 3 occurrences. Servers MAY track total reconsider-then-re-decline events per user and apply additional rate limits or reputation penalties for excessive cycling.

## 20. Reserved Operations

The following operation names are reserved for future use. Implementations MUST NOT use these names for custom extensions.

**`schelling.calibrate`.** Intended for anti-gaming calibration checks and cross-agent calibration alignment. Will allow a server to: (a) verify that an agent's embedding generation is consistent and not manipulated, (b) provide reference embeddings for common intents so agents can calibrate against a standard, and (c) track cross-agent-model matching quality and flag when certain model pairs have systematically worse outcomes than others.

**`schelling.federate`.** Intended for federation negotiation between Schelling server nodes.

**`schelling.group_search`.** Reserved for future N-party group matching. Would accept a target group size and return compatible groups rather than individual candidates.

## 21. Coordination Kernel Extensions

Version 2.0 extends the Schelling Protocol from a matching protocol to a coordination kernel. The five extensions in this section turn the protocol into a substrate for ongoing agent-to-agent coordination — not just finding matches, but negotiating, agreeing, and collaborating after the match is made.

**Design principle: the server stays dumb.** All five features follow the same division of labor as the rest of the protocol. The server stores data, does math (cosine similarity, exact-match filtering), and relays messages. It does not interpret content, parse natural language, or make intelligent decisions. All intelligence lives in agents.

**Independence.** Each feature is independently valuable and can be implemented in any order. They integrate with the existing stage system, reputation, and privacy layers but do not depend on each other.

### 21.1 Pre-Commitment Agent Dialogue (`schelling.inquire`)

#### 21.1.1 Overview

`schelling.inquire` enables structured question-and-answer exchanges between agents at the EVALUATED stage — after agents have each other's embeddings and compatibility scores, but before either party commits. This fills the gap between discovery and commitment: the "due diligence" step where agents clarify deal-breakers, logistics, and expectations before recommending commitment to their users.

The server relays JSON blobs between agents. It does not interpret, validate, or moderate the content of questions or answers. Rate limiting prevents interrogation.

#### 21.1.2 Operation: `schelling.inquire`

**Group:** Coordination. **Authentication:** Bearer token.

Send a question to, or answer a question from, the other party in a candidate pair.

**Input fields (sending a question):**

| Field | Type | Required | Description |
|---|---|---|---|
| `user_token` | string | Yes | Caller's bearer token. |
| `candidate_id` | string | Yes | The candidate group ID. |
| `action` | string | Yes | `"ask"`. |
| `question` | string | Yes | The question text. Maximum 2,000 characters. |
| `category` | string | No | Question category (e.g., `"dealbreakers"`, `"logistics"`, `"compensation"`, `"lifestyle"`, `"custom"`). For display and analytics. |
| `required` | boolean | No | If `true`, the asking agent considers an answer necessary before committing. Default: `false`. |
| `idempotency_key` | string | No | Idempotency key. |

**Input fields (answering a question):**

| Field | Type | Required | Description |
|---|---|---|---|
| `user_token` | string | Yes | Caller's bearer token. |
| `candidate_id` | string | Yes | The candidate group ID. |
| `action` | string | Yes | `"answer"`. |
| `inquiry_id` | string | Yes | The ID of the question being answered. |
| `answer` | string | Yes | The answer text. Maximum 2,000 characters. |
| `confidence` | float | No | Confidence in the answer, in [0, 1]. Default: 1.0. |

- Q&A history accumulates on the candidate record and is available to both agents via `action: "list"`.
- Agents use Q&A history to inform their commit/decline decisions. The server does not interpret or score the content.
- Questions without answers after 7 days are marked as expired (still visible in history but flagged).
- The `required` flag is informational — it signals intent to the other agent but does not enforce any behavior.

### 21.2 Push-Based Discovery (`schelling.subscribe`)

#### 21.2.1 Overview

`schelling.subscribe` enables event-driven discovery by registering standing queries. Instead of polling `schelling.search` repeatedly, an agent registers a subscription with an intent embedding, optional hard filters, and a similarity threshold. When a new user registers whose profile exceeds the threshold, the server stores a notification for the subscribing agent.

This is not a push notification system — the server does not call the agent. Instead, notifications accumulate and are retrieved via `schelling.notifications`. This preserves the protocol's poll-based architecture (§13.5, §16.4) while enabling event-driven discovery.

#### 21.2.2 Operation: `schelling.subscribe`

**Group:** Discovery. **Authentication:** Bearer token.

Register a standing discovery query.

**Input fields:**

| Field | Type | Required | Description |
|---|---|---|---|
| `user_token` | string | Yes | Caller's bearer token. |
| `intent_embedding` | array of 16 floats | Yes | The intent embedding to match against new registrations. Typically the subscriber's own intent embedding. |
| `threshold` | float | Yes | Minimum combined score for a notification. Range: 0–1. |
| `hard_filters` | object | No | Structured attribute filters (same format as `schelling.search` hard_filters). |
| `capability_filters` | array of strings | No | Agent capability filters (same format as `schelling.search` capability_filters). |
| `max_notifications_per_day` | integer | No | Maximum notifications per day. Range: 1–50. Default: 10. |
| `ttl_days` | integer | No | Subscription time-to-live in days. Range: 1–90. Default: 30. |
| `idempotency_key` | string | No | Idempotency key. |

**Output fields:**

| Field | Type | Description |
|---|---|---|
| `subscription_id` | string | Opaque identifier for the subscription. |
| `created_at` | string | ISO 8601 timestamp. |
| `expires_at` | string | ISO 8601 timestamp when the subscription expires. |

**Error codes:** `USER_NOT_FOUND`, `INVALID_INPUT`, `INVALID_INTENT_EMBEDDING`, `RATE_LIMITED`, `MAX_SUBSCRIPTIONS`.

**Rate limiting:** Maximum 10 active subscriptions per user.

#### 21.2.3 Operation: `schelling.unsubscribe`

**Group:** Discovery. **Authentication:** Bearer token.

Cancel an active subscription.

**Input fields:**

| Field | Type | Required | Description |
|---|---|---|---|
| `user_token` | string | Yes | Caller's bearer token. |
| `subscription_id` | string | Yes | The subscription to cancel. |

**Output fields:**

| Field | Type | Description |
|---|---|---|
| `cancelled` | boolean | Always `true`. |

**Error codes:** `USER_NOT_FOUND`, `SUBSCRIPTION_NOT_FOUND`.

#### 21.2.4 Operation: `schelling.notifications`

**Group:** Discovery. **Authentication:** Bearer token.

Retrieve accumulated subscription notifications.

**Input fields:**

| Field | Type | Required | Description |
|---|---|---|---|
| `user_token` | string | Yes | Caller's bearer token. |
| `subscription_id` | string | No | Filter to a specific subscription. If omitted, return all. |
| `since` | string | No | Return only notifications after this ISO 8601 timestamp. |
| `limit` | integer | No | Maximum notifications. Range: 1–100. Default: 50. |

**Output fields:**

| Field | Type | Description |
|---|---|---|
| `notifications` | array | Subscription match notifications. |
| `notifications[].notification_id` | string | Notification ID. |
| `notifications[].subscription_id` | string | The triggering subscription. |
| `notifications[].candidate_token_hash` | string | Opaque hash of the matched user (not their real token). Use `schelling.search` to find them. |
| `notifications[].combined_score` | float | Estimated combined score (quantized to 2 decimal places). |
| `notifications[].intent_similarity` | float | Intent similarity (quantized to 2 decimal places). |
| `notifications[].matched_at` | string | ISO 8601 timestamp. |
| `total` | integer | Total unread notifications. |

**Error codes:** `USER_NOT_FOUND`, `SUBSCRIPTION_NOT_FOUND`.

**Server-side behavior:** When a new user registers (via `schelling.register`), the server evaluates the new registration against all active subscriptions:
1. Check hard_filters (exact match, same as search).
2. Compute combined score using the subscription's intent embedding and the subscriber's stored trait embedding against the new user's embeddings.
3. If the combined score exceeds the subscription's threshold, create a notification.
4. Respect `max_notifications_per_day` — if the daily cap is reached, skip.

**Cost:** Trivial. The server stores the subscription's intent embedding and filters. On each new registration, it iterates active subscriptions (expected to be a small number per-user) and computes cosine similarity — the same math it already does for search.

**Privacy:** Notifications reveal only a hashed identifier and quantized scores. The subscriber must call `schelling.search` to discover the full candidate record through the normal funnel.

### 21.3 Agent Capability Layer

#### 21.3.1 Overview

The `agent_capabilities` field extends `schelling.register` with structured descriptions of what the *agent* can do — not who the user is. This enables agent-to-agent task routing: an agent looking for "someone whose agent can schedule meetings" can filter by capability without relying on embedding similarity.

Capabilities are visible at stage 1 (DISCOVERED), the same as structured attributes. They use the same exact-match filtering mechanism.

#### 21.3.2 Registration Extension

The `schelling.register` and `schelling.update` operations accept an optional `agent_capabilities` field:

| Field | Type | Required | Description |
|---|---|---|---|
| `agent_capabilities` | array of objects | No | Structured description of what the agent can do. Each entry: `{capability: string, parameters: object, confidence: float}`. |

**Capability object schema:**

| Field | Type | Required | Description |
|---|---|---|---|
| `capability` | string | Yes | Machine-readable capability identifier (snake_case). Examples: `"can_schedule_meetings"`, `"can_process_payments"`, `"speaks_languages"`, `"has_calendar_access"`, `"can_video_call"`, `"can_sign_documents"`. |
| `parameters` | object | No | Capability-specific parameters. For `"speaks_languages"`: `{"languages": ["en", "es", "zh"]}`. For `"can_process_payments"`: `{"methods": ["stripe", "paypal"]}`. |
| `confidence` | float | No | Agent's confidence in this capability, in [0, 1]. Default: 1.0. |

**Example registration:**

```json
{
  "intent_embedding": [0.82, 0.75, ...],
  "intents": ["find a business partner"],
  "embedding": [0.3, -0.5, ...],
  "agent_capabilities": [
    {"capability": "can_schedule_meetings", "parameters": {"calendars": ["google", "outlook"]}, "confidence": 1.0},
    {"capability": "speaks_languages", "parameters": {"languages": ["en", "es", "zh"]}, "confidence": 0.9},
    {"capability": "can_process_payments", "parameters": {"methods": ["stripe"]}, "confidence": 1.0},
    {"capability": "has_calendar_access", "confidence": 1.0}
  ]
}
```

**Validation:** Maximum 20 capabilities per registration. Capability identifiers must be snake_case, 1–100 characters. Parameters object must be JSON-serializable, maximum 1KB per capability.

#### 21.3.3 Search Extension

`schelling.search` accepts an optional `capability_filters` parameter:

| Field | Type | Required | Description |
|---|---|---|---|
| `capability_filters` | array of strings | No | Required capability identifiers. Candidates must have ALL listed capabilities. Conjunctive/AND semantics, same as `hard_filters`. |

**Example search:**

```json
{
  "user_token": "...",
  "capability_filters": ["can_schedule_meetings", "speaks_languages"],
  "hard_filters": {"languages": "zh"}
}
```

This returns only candidates whose agents declare both `can_schedule_meetings` and `speaks_languages` capabilities, AND whose structured attributes include `"zh"` in their languages.

**Privacy:** Agent capabilities are visible at stage 1 (DISCOVERED). They are included in search results and candidate records alongside structured attributes. Capabilities describe agent software features, not personal user data, so they carry minimal privacy risk.

**Output:** Search results include `agent_capabilities` in each candidate record (at all stages).

### 21.4 Post-Commitment Structured Agreements (`schelling.contract`)

#### 21.4.5 Reputation Integration

Contracts tie into the reputation system:
- **Completed contracts** (`action: "complete"` by both parties) create a positive reputation event for both parties (equivalent weight to a `"positive"` outcome report).
- **Terminated contracts** create a negative reputation event for the terminating party (weight 0.5 of a `"negative"` outcome report), unless both parties terminate (mutual dissolution — no reputation impact).
- **Expired contracts** (status transitions to `"expired"` without completion) create a minor negative signal for both parties (weight 0.25 of a `"negative"` outcome).
- Reputation events from contracts are tagged with `event_source: "contract"` and include the contract type for cluster-specific analytics.

#### 21.5.3 Reputation Integration

Events feed into reputation indirectly:
- **Completion events** on contracts (type `"completion"`) are positive signals, weighted the same as explicit `schelling.report` positive outcomes.
- **Unacknowledged events** (where `requires_ack` is true and the deadline passes) create a minor negative signal for the non-acknowledging party (weight 0.1).
- **Issue events** do not directly affect reputation. They serve as documentation that may be referenced in disputes.
- The count of events on a contract contributes to the "engagement" signal: contracts with regular events demonstrate active coordination, which is a positive reputation indicator.

| `contract_amendment` | `schelling.contract_update` | A contract amendment has been proposed. |

### 22.7 Updated Cluster Configuration

Clusters MAY include the following optional configuration fields:

| Field | Type | Required | Description |
|---|---|---|---|
| `suggested_inquiry_categories` | array of strings | No | Suggested question categories for the inquiry system. |
| `default_contract_ttl_days` | integer | No | Default contract expiry. Default: 30. |
| `max_active_contracts` | integer | No | Maximum active contracts per candidate pair. Default: 5. |

### 22.8 Data Export and Deletion

`schelling.export` (§5.27) is extended to include:
- `user_data.inquiries` — All inquiry Q&A records.
- `user_data.subscriptions` — All subscription records (active and expired).
- `user_data.contracts` — All contract records.
- `user_data.events` — All lifecycle event records.

`schelling.delete_account` (§5.28) cascade is extended:
- Inquiries — deleted.
- Subscriptions and notifications — deleted.
- Contracts — anonymized (contract records preserved with `"DELETED_USER"` sentinel, so the other party's contract history remains intact).
- Events — deleted.
